## 1. 核心判断

AI 产品经理工作台不能只做成：

```text
一句话 → PRD
一句话 → 竞品报告
一句话 → HTML 原型
```

这种模式容易变成一次性 Prompt 工具，短期有新鲜感，但很难长期进入产品经理日常工作。

真正有价值的方向是：

> 以产品资产为中心，让 AI 帮助产品经理持续生产、编辑、评审、追踪和复用产品资产。

因此本模块的长期定位不是“AI 产品经理聊天机器人”，而是：

> **Product Asset Workspace / 产品资产工作台**

AI 是入口和助手，核心资产包括：

- 需求；
- PRD；
- 用户故事；
- 验收标准；
- 竞品档案；
- 竞品报告；
- 功能矩阵；
- 原型规格；
- 页面流程；
- 交互注释；
- 研发任务；
- 评审记录；
- 版本变更；
- 上线复盘。

## 2. 产品定位

### 2.1 不是单点 AI 工具

不应该只解决“生成一份文档”或“生成一个页面”的问题。

原因：

- 产品经理需要持续修改；
- 需求需要多人评审；
- 设计需要注释和状态说明；
- 研发需要边界、验收和异常逻辑；
- 竞品和用户反馈会持续变化；
- 产物需要版本管理和追踪。

### 2.2 是产品资产工作台

推荐定位：

```text
产品经理把想法、资料、竞品、用户反馈和研发约束输入平台；
平台通过 AI 和结构化工作流，把它们沉淀为可编辑、可评审、可版本化、可复用的产品资产。
```

### 2.3 与传统工具的关系

AI 产品经理工作台不是一开始替代 Axure、Figma、飞书、Notion、Confluence、Jira。

更现实的路径是：

```text
先成为产品资产生成与评审中心
  ↓
再成为产品上下文和知识中心
  ↓
再连接原型、研发任务、外部工具
  ↓
最后逐步承接部分设计和交付流程
```

## 3. 总体架构

```text
AI Product Workspace
├── Product Space 产品空间
├── File Center 文件中心
├── Context Hub 上下文中心
├── Artifact System 产物系统
├── Agent Workflow 智能体工作流
├── PRD Editor 文档编辑器
├── Prototype System 原型系统
├── Review & Delivery 评审交付系统
└── Integration Layer 外部集成层
```

整体关系：

```text
用户输入 / 文件 / 历史资产 / 外部资料
  ↓
File Center 保存原始资料
  ↓
Context Hub 解析、结构化、索引
  ↓
Agent Workflow 澄清、分析、生成、评审
  ↓
Artifact System 沉淀 PRD / 报告 / 原型规格
  ↓
Editor / Prototype / Review 持续编辑和评审
  ↓
Delivery 输出研发任务、导出文档、同步外部系统
```

当前 Terminal Agent 是执行型 Agent：

```text

  ↓

  ↓

  ↓
```


## 5. Product Space 产品空间

### 5.1 作用

Product Space 是产品资产的顶层容器。

一个租户可以有多个产品空间：

```text
租户：OpsinTech
├── 产品空间：智能运维平台
├── 产品空间：AI 产品经理工作台
├── 产品空间：智能终端
└── 产品空间：告警平台
```

每个产品空间下聚合：

- 文件；
- PRD；
- 竞品；
- 原型；
- 任务；
- 知识；
- 评审；
- 版本。

### 5.2 设计目标

Product Space 解决两个问题：

1. 给 AI 一个明确的上下文边界；
2. 给用户一个长期维护产品资产的工作区。

### 5.3 建议数据模型

```text
product_spaces
  id
  tenant_id
  name
  description
  owner_user_id
  status: active | archived
  visibility: private | tenant
  created_by
  created_at
  updated_at
```

```text
product_space_members
  id
  tenant_id
  product_space_id
  user_id
  role: owner | editor | viewer
  created_at
```

```text
product_modules
  id
  tenant_id
  product_space_id
  name
  description
  parent_id
  created_at
  updated_at
```

## 6. File Center 文件中心

### 6.1 作用

File Center 负责保存原始资料和 AI 生成文件。

包括：

- 用户上传文件；
- 租户公共文件；
- AI 生成文件；
- 业务附件；
- 原型导入物；
- 竞品截图；
- 会议纪要；
- 历史 PRD。

详细规划参考：

[用户与租户文件空间版本规划](file:///Users/kevinliangx/Developer/Repos/PublicCodeHub/KevinLiangX/opsintech/opsintech-platform/产品规划/版本管理/2026-06-19_用户与租户文件空间版本规划.md)

### 6.2 与产品工作台关系

```text
File Center 保存文件
  ↓
Context Hub 解析文件
  ↓
Agent 使用文件内容
  ↓
Artifact 引用文件来源
```

### 6.3 关键原则

- AI 生成文件默认归属用户私有空间；
- 用户确认后才能发布到租户空间；
- 文件权限必须在 AI 检索和预览阶段统一生效；
- 不直接依赖本地目录作为业务权限；
- 后续通过 StorageService 支持 MinIO。

## 7. Context Hub 上下文中心

### 7.1 作用

Context Hub 是 AI 真正“懂产品”的基础。

它负责把分散的资料转成 AI 可用上下文：

```text
文件 / PRD / 会议纪要 / 竞品资料 / 用户反馈
  ↓
解析
  ↓
结构化
  ↓
标注来源
  ↓
权限过滤
  ↓
供 Agent 使用
```

### 7.2 上下文类型

```text
Product Context
├── 产品基本信息
├── 模块结构
├── 用户角色
├── 业务术语
├── 历史 PRD
├── 当前 Roadmap
├── 竞品资料
├── 用户反馈
├── 设计资料
├── 研发约束
└── 评审结论
```

### 7.3 第一版策略

第一版不建议默认全量向量化。

推荐：

```text
用户显式选择文件 / 文档
  ↓
AI 读取解析文本
  ↓
输出中标注来源
```

后续再支持：

```text
选择知识库范围
  ↓
关键词检索
  ↓
可选向量化
  ↓
混合检索
```

### 7.4 权限原则

AI 检索必须先做权限过滤，再召回内容。

错误方式：

```text
全库检索 top-k
  ↓
应用层过滤无权限文档
```

正确方式：

```text
按 tenant_id / owner_user_id / visibility / product_space_id 构造查询范围
  ↓
只在授权范围内检索
  ↓
返回可引用内容
```

## 9. PRD Editor 文档编辑器

### 9.1 目标

PRD Editor 不是简单 Markdown 编辑器，而是结构化产品文档编辑器。

用户需要：

- 改标题；
- 改背景；
- 增删用户故事；
- 调整验收标准；
- 评论某段内容；
- 对某一节让 AI 重写；
- 对比版本；
- 导出文档；
- 关联原型和任务。

### 9.2 推荐交互

```text
左侧：文档目录 / 章节导航
中间：结构化文档编辑区
右侧：AI 助手 / 来源引用 / 评论 / 版本
顶部：状态 / 保存 / 导出 / 发起评审
```

### 9.3 AI 操作粒度

AI 不应该只能“全文重写”，还应该支持局部操作：

```text
帮我补充这一节的验收标准
帮我把这个用户故事拆成 3 个
帮我检查这个需求的边界条件
帮我根据这些评论修改当前章节
帮我把这段话改得更适合研发评审
```

### 9.4 质量校验

PRD 保存或评审前，AI 可以检查：

- 是否有明确目标用户；
- 是否有业务目标；
- 是否有成功指标；
- 是否有范围边界；
- 是否有非功能需求；
- 是否有异常状态；
- 是否有权限说明；
- 是否有验收标准；
- 是否存在待确认问题。

## 10. Prototype System 原型系统

### 10.1 核心判断

原型系统不能以 HTML 作为源产物。

HTML 的问题：

- 用户不好编辑；
- 注释难管理；
- 页面结构和产品意图不清晰；
- 交互逻辑散落在代码里；
- 不易版本 diff；
- 不易关联 PRD；
- 不适合长期产品资产沉淀。

正确方向：

```text
结构化原型模型是源产物
HTML / Canvas / 图片只是渲染结果
```

### 10.2 原型能力分层

#### 第一层：原型说明

从 PRD 生成：

- 页面清单；
- 用户流程；
- 字段定义；
- 页面状态；
- 交互说明；
- 异常情况；
- 权限状态；
- 埋点建议；
- 研发说明。

这一层最容易落地，也最贴近日常评审。

#### 第二层：结构化线框稿

生成页面 DSL / JSON，而不是直接生成 HTML。

示例：

```json
{
  "page": "FileCenterHome",
  "layout": "sidebar-content-detail",
  "regions": [
    {
      "id": "sidebar",
      "type": "navigation",
      "items": ["我的文件", "租户空间", "AI产物", "回收站"]
    },
    {
      "id": "toolbar",
      "type": "actions",
      "items": ["上传", "新建文件夹", "搜索"]
    },
    {
      "id": "file_list",
      "type": "table",
      "columns": ["名称", "大小", "类型", "更新时间", "操作"]
    }
  ],
  "annotations": [
    {
      "target": "upload_button",
      "text": "上传前需要检查租户配额和用户权限"
    }
  ]
}
```

#### 第三层：可编辑画布

后续建设轻量可编辑画布：

```text
Canvas Editor
├── 页面树
├── 组件面板
├── 属性面板
├── 交互面板
├── 注释面板
├── 状态面板
└── AI 修改面板
```

#### 第四层：已有设计识别

支持导入或识别：

- 截图；
- HTML 原型；
- Axure HTML 导出物；
- Figma 文件或 API；
- 历史 PRD + 设计说明。

第一阶段不要追求完整还原 Axure，而是追求：

```text
已有设计
  ↓
识别页面结构
  ↓
生成页面说明
  ↓
补充交互注释
  ↓
关联 PRD
```

## 11. Agent Workflow 智能体工作流

### 11.1 Requirement Agent

目标：从想法到可评审 PRD。

工作流：

```text
输入想法 / 资料
  ↓
识别需求类型
  ↓
主动澄清问题
  ↓
生成需求分析
  ↓
生成 PRD Artifact
  ↓
生成用户故事和验收标准
  ↓
AI 自检
  ↓
保存版本
```

核心工具：

- `load_product_context`；
- `ask_clarifying_questions`；
- `generate_requirement_analysis`；
- `generate_prd_artifact`；
- `validate_prd`；
- `save_artifact_version`。

### 11.2 Competitor Agent

目标：从竞品资料到可追踪竞品资产。

工作流：

```text
配置竞品
  ↓
录入官网 / 文档 / 定价 / Release Note
  ↓
抓取或上传资料
  ↓
生成竞品档案
  ↓
生成功能矩阵
  ↓
生成报告 / Battlecard
  ↓
持续更新变化
```

第一版可以先人工触发，不必立即做定时监控。

### 11.3 Prototype Agent

目标：从 PRD 到原型规格。

工作流：

```text
选择 PRD
  ↓
识别用户流程
  ↓
生成页面清单
  ↓
生成页面结构
  ↓
生成交互注释
  ↓
生成低保真 DSL
  ↓
保存 Prototype Spec
```

### 11.4 Review Agent

目标：帮助评审和发现遗漏。

可检查：

- PRD 是否完整；
- 原型是否覆盖 PRD；
- 验收标准是否可测试；
- 是否缺少异常状态；
- 是否缺少权限说明；
- 是否存在逻辑矛盾；
- 是否可以拆解研发任务。

## 17. 成功指标

### 17.1 用户价值指标

- 用户是否愿意基于 AI 初稿继续编辑；
- PRD 起草时间是否减少 50%；
- 评审前需求遗漏是否减少；
- 研发对需求清晰度评分是否提升；
- 用户是否愿意把最终稿沉淀在平台，而不是复制到别处。

- 每个产品空间的 Artifact 数量；
- PRD 从创建到评审的完成率；
- AI 生成后被用户编辑的比例；
- 文档版本数量；
- 来源引用使用率；

## 18. 关键风险

### 18.1 做成一次性生成工具

风险：用户试一次觉得有趣，但不会长期维护。

应对：必须强调 Artifact、编辑、版本、评论和评审。

### 18.2 原型过早做重

风险：一开始做类似 Axure / Figma 的画布，开发成本高，短期价值不稳定。

应对：先做原型说明和结构化 DSL，再做低保真预览，最后做可编辑画布。

### 18.3 AI 输出没有上下文

风险：生成内容泛泛而谈，不符合公司业务。

应对：接入 File Center 和 Context Hub，先支持用户显式选择资料。

### 18.4 权限和隐私风险

风险：AI 读取了用户不该访问的文件，或者私有草稿进入租户知识库。

应对：所有文件、Artifact、检索都必须带 `tenant_id` 和权限过滤。

### 18.5 只导出，不沉淀

风险：用户把内容复制到飞书、Notion、Axure 后，平台不再是资产中心。

应对：平台内必须提供足够好的编辑、评审、版本和引用能力。

## 19. 推荐下一步

建议下一步不要直接做原型或竞品，而是先拆 V0.1 技术实施方案。

优先事项：

1. 定义 Product Space 数据模型；
2. 定义 PRD Artifact Schema；
3. 定义 Artifact Version 模型；
4. 设计 Requirement Agent 工作流；
5. 设计 PRD Editor MVP；
6. 设计 AI 澄清问题和 PRD 生成 Prompt；
7. 接入文件中心中的用户私有 AI 产物保存；
8. 完成“想法 → 澄清 → PRD → 编辑 → 保存版本”的闭环。

第一阶段的验收标准：

```text
用户可以在一个产品空间中输入一句需求想法；
AI 能主动提出澄清问题；
用户回答后生成结构化 PRD；
PRD 可以编辑、保存版本、导出 Markdown；
AI 生成的 PRD 作为用户私有 Artifact 存在；
后续可以发布到租户空间或进入评审。
```
