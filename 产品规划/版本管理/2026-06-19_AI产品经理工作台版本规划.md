创建日期：2026-06-19

## 1. 背景

产品经理日常工作包含需求分析、竞品分析、原型设计、研发协作、上线复盘等多个环节。

当前 AI 工具已经能够辅助完成部分单点任务，例如：

- 根据一句想法生成 PRD 草稿；
- 总结会议纪要；
- 检索竞品信息；
- 生成用户故事和验收标准；
- 快速生成界面原型或可交互 Demo。

但现有工具普遍存在一个问题：

> 多数 AI 工具停留在“单次生成”或“文档助手”层面，缺少企业自身上下文、缺少证据链、缺少持续版本管理，也难以和产品研发流程形成闭环。

OpsinTech 平台当前已经具备 Agent 对话、工具调用、历史会话、任务执行、前后端协作等基础能力，因此可以在现有平台上扩展出面向产品经理的 AI 工作台。

---

## 2. 产品定位

AI 产品经理工作台不是一个简单的聊天机器人，也不是单纯的 PRD 生成器。

它的定位是：

> 帮助产品经理把想法、资料、竞品信息、用户反馈和研发约束转化为可沉淀、可追踪、可协作的产品资产。

核心目标：

- 降低需求分析和文档整理成本；
- 提高 PRD、竞品报告、原型说明的质量；
- 让 AI 输出具备业务上下文和证据来源；
- 将产品工作从“临时聊天”升级为“持续工作流”；
- 最终形成从想法到需求、竞品、原型、任务、复盘的闭环。

## 4. 核心设计原则

### 4.1 Chat 是入口，不是最终形态

用户可以通过对话表达想法，但最终需要沉淀为结构化资产，例如：

- PRD；
- 竞品分析报告；
- 功能矩阵；
- 原型说明；
- 用户故事；
- 验收标准；
- 研发任务；
- 版本路线图。

### 4.2 AI 输出必须有证据来源

AI 给出的结论应尽可能引用来源：

- 用户反馈；
- 客服工单；
- 历史 PRD；
- 会议纪要；
- 竞品页面；
- 数据指标；
- 研发约束；
- 产品路线图。

没有证据链的 AI 结论只能作为草稿，不能直接作为产品决策依据。

### 4.3 AI 应主动追问

优秀的需求分析不是直接生成文档，而是先澄清关键问题：

- 目标用户是谁？
- 解决什么真实问题？
- 成功指标是什么？
- 这是增长需求、效率需求还是风控需求？
- 是否已有类似能力？
- 是否有上线时间要求？
- 是否有技术、合规、数据限制？

### 4.4 产物必须可编辑、可版本化

AI 生成内容不是终稿。

产品经理需要能够：

- 编辑文档；
- 局部重写；
- 对比历史版本；
- 评论协作；
- 导出分享；
- 关联研发任务；
- 追踪后续变更。

### 4.5 从单次生成走向持续工作流

目标工作流：

```text
想法
  ↓
需求澄清
  ↓
PRD 草稿
  ↓
竞品对比
  ↓
原型设计
  ↓
研发任务拆解
  ↓
评审协作
  ↓
上线复盘
```

## 5. 版本路线图

## V0.1：需求分析 MVP

### 版本目标

验证产品经理是否愿意使用 AI 完成需求分析和 PRD 初稿生成。

### 核心用户

- 产品经理；
- 创业团队负责人；
- 研发负责人；
- 需要快速梳理需求的业务人员。

### 核心场景

用户输入一句需求想法，例如：

```text
我想做一个 AI 竞品分析功能，帮产品经理自动追踪竞品变化。
```

系统输出结构化需求分析文档。

### 功能范围

#### 1. 需求输入

支持用户输入：

- 一句话想法；
- 老板需求；
- 客户反馈；
- 会议纪要；
- 粗略功能描述。

#### 2. AI 需求澄清

AI 根据输入主动提出澄清问题：

- 目标用户；
- 业务目标；
- 成功指标；
- 约束条件；
- 优先级；
- 上线时间；
- 依赖关系。

#### 3. PRD 草稿生成

生成内容包括：

- 需求背景；
- 用户画像；
- 核心场景；
- 用户痛点；
- 业务目标；
- 功能范围；
- 非功能需求；
- 边界条件；
- 风险点；
- 验收标准；
- 待确认问题。

#### 4. 用户故事生成

按照以下格式生成：

```text
作为一个 [用户角色]
我希望 [完成某个动作]
以便 [获得某种价值]
```

#### 5. 验收标准生成

输出 Given / When / Then 或清单式验收标准。

#### 6. 历史会话与文档保存

支持保存：

- 需求会话；
- 生成文档；
- 用户修改记录；
- 当前版本状态。

### 暂不包含

- 复杂知识库 RAG；
- 自动抓取竞品；
- Figma 级原型；
- 与 Jira / Linear 同步；
- 多人实时协作。

### 成功标准

- 用户可以在 5 分钟内从一句想法得到一份可讨论的 PRD 草稿；
- AI 至少能提出 5 个关键澄清问题；
- 生成的需求文档结构稳定；
- 用户愿意基于 AI 初稿继续编辑，而不是完全重写。

## V0.2：产品知识库增强

### 版本目标

让 AI 从“通用产品助手”升级为“懂公司业务的产品助手”。

### 核心能力

#### 1. 文档上传

支持上传：

- 历史 PRD；
- 用户反馈；
- 客服工单；
- 销售记录；
- 会议纪要；
- 设计说明；
- 发布记录；
- 技术文档。

#### 2. 知识索引

建立 Product Context Hub：

- 文档切分；
- 向量索引；
- 关键词索引；
- 标签分类；
- 时间线归档。

#### 3. 引用溯源

AI 输出时标注来源：

```text
该结论参考：
- 2026-05-12 客户访谈记录
- 2026-04-30 告警模块 PRD
- 2026-03-31 智能运维平台方案与竞品分析
```

#### 4. 产品术语库

沉淀内部术语：

- 模块名称；
- 用户角色；
- 业务指标；
- 产品能力；
- 技术边界。

### 成功标准

- AI 能正确引用历史文档；
- AI 输出中能体现公司已有产品上下文；
- 用户可以基于知识库问答快速了解历史决策；
- PRD 草稿不再是泛泛而谈。

## V0.3：竞品分析 Agent

### 版本目标

建立持续竞品跟踪能力，帮助产品经理形成市场洞察。

### 核心场景

用户创建一个竞品空间：

```text
竞品：
- Linear
- Notion
- Jira
- ClickUp
- Asana

关注维度：
- AI 功能
- 定价
- 协作能力
- 自动化
- 产品定位
```

系统定期生成竞品分析报告。

### 核心能力

#### 1. 竞品档案

记录：

- 公司简介；
- 产品定位；
- 目标用户；
- 核心功能；
- 定价模式；
- 官网链接；
- 文档链接；
- Release Note 链接。

#### 2. 功能矩阵

对比多个竞品在不同维度上的能力：

| 功能维度 | 我们 | 竞品 A | 竞品 B | 机会点 |
|---|---|---|---|---|
| AI 需求分析 | 规划中 | 已支持 | 部分支持 | 可优先切入 |
| 竞品监控 | 规划中 | 不明显 | 已支持 | 形成差异化 |

#### 3. 变化监控

定期跟踪：

- 官网文案变化；
- 定价变化；
- 新功能发布；
- 博客更新；
- 招聘信息；
- 文档变更；
- 用户评价变化。

#### 4. 竞品 Battlecard

面向内部销售、产品、管理层输出：

- 竞品优势；
- 竞品劣势；
- 我们的差异化；
- 应对话术；
- 风险提示。

#### 5. 周报 / 月报

自动生成：

- 本周竞品动态；
- 重要变化；
- 市场趋势；
- 建议关注事项；
- 对产品路线图的影响。

### 成功标准

- 用户可以配置竞品和关注维度；
- 系统能生成结构化竞品报告；
- 关键结论有来源链接；
- 至少支持人工触发分析，后续再扩展定时监控。

## V0.4：原型设计 Agent

### 版本目标

从 PRD 延伸到原型设计，帮助产品经理快速表达产品方案。

### 能力分层

#### 第一层：文本原型

输出：

- 页面结构；
- 用户流程；
- 字段定义；
- 状态流转；
- 空状态；
- 错误状态；
- 权限状态；
- 信息架构。

#### 第二层：低保真原型

输出：

- Markdown 原型；
- Mermaid 流程图；
- 页面线框描述；
- 交互路径说明。

#### 第三层：可交互 Demo

生成可运行的前端 Demo：

- HTML / CSS / JavaScript；
- React / Next.js 组件草稿；
- 基于现有设计系统的页面；
- 支持基本点击和状态切换。

### 原型 Agent 需要回答的问题

- 页面解决什么问题？
- 用户第一眼看到什么？
- 主操作是什么？
- 次要操作是什么？
- 空状态如何展示？
- 错误状态如何展示？
- 移动端如何适配？
- 这个设计如何支撑业务指标？

### 成功标准

- 用户可以从 PRD 生成页面结构；
- 用户可以快速得到低保真原型；
- 原型说明能辅助设计和研发评审；
- 可交互 Demo 可作为后续增强方向。

## V0.5：产品交付闭环

### 版本目标

把需求、竞品、原型进一步连接到研发协作和上线复盘。

### 核心能力

#### 1. 研发任务拆解

从 PRD 自动拆解：

- Epic；
- Story；
- Task；
- Bug；
- 测试用例；
- 验收清单。

#### 2. 外部系统同步

可选集成：

- GitHub Issues；
- Linear；
- Jira；
- 飞书；
- Notion；
- Confluence。

#### 3. 评审纪要

支持产品评审、技术评审后的：

- 会议纪要生成；
- 决策记录；
- 风险记录；
- 待办事项；
- 需求变更记录。

#### 4. 发布说明

根据需求和任务生成：

- Release Note；
- 用户公告；
- 内部培训材料；
- FAQ。

#### 5. 上线复盘

沉淀：

- 实际效果；
- 数据指标；
- 用户反馈；
- 问题复盘；
- 下一轮优化建议。

### 成功标准

- 从 PRD 到研发任务可以形成半自动链路；
- 产品变更有记录；
- 上线后能回收反馈；
- 产品资产可以持续复用。

## 6. 推荐实施顺序

### 第一阶段：先做需求分析 MVP

原因：

- 离产品经理日常最近；
- 使用频率高；
- 产物明确；
- 技术复杂度可控；
- 可以复用当前 Agent UI 和 Prompt 架构；
- 后续自然连接知识库、竞品和原型。

### 第二阶段：补产品知识库

原因：

- 没有上下文，AI 输出容易泛化；
- 知识库是后续所有 Agent 的基础；
- 需求分析、竞品分析、原型设计都依赖上下文。

### 第三阶段：做竞品分析 Agent

原因：

- 竞品分析适合 Agent 化；
- 可以持续监控；
- 能形成差异化价值；
- 对产品规划和战略判断有帮助。

### 第四阶段：做原型设计 Agent

原因：

- 原型质量要求高；
- 需要设计系统和前端能力支撑；
- 适合在 PRD 结构稳定后再做。

### 第五阶段：做交付闭环

原因：

- 需要和研发工具、项目管理系统连接；
- 需要前面已有稳定的 PRD 和任务结构；
- 是平台长期价值的体现。

## 9. 当前结论

第一阶段不建议同时做需求分析、竞品分析、原型设计三个完整模块。

推荐版本切入点：

> V0.1 先做 AI 需求分析工作台，通过一句话需求生成结构化 PRD 草稿，并支持 AI 主动澄清、用户编辑和历史保存。

后续再基于需求分析沉淀出来的结构化资产，逐步扩展：

```text
需求分析
  ↓
产品知识库
  ↓
竞品分析
  ↓
原型设计
  ↓
研发交付闭环
```

最终目标是形成一个真正服务产品经理日常工作的 AI Product Workspace，而不是单点 Prompt 工具。
