# 产品资产智能工作台：设计规范与产品知识库方案

> 日期：2026-06-19  
> 状态：规划草案  
> 适用范围：OpsinTech 平台的产品规划、需求分析、PRD 编写、页面结构设计与后续原型生成能力  
> 核心定位：不是通用 AI 产品经理，而是基于已有产品资产、设计规范和业务规则的受约束产品规划系统。

---

## 1. 背景

当前产品规划工作存在明显割裂：

- 日常想法和讨论记录在腾讯文档、本地 Markdown、聊天记录中；
- 版本规划、需求分析、PRD、技术方案之间缺少统一关联；
- AI 可以快速生成内容，但多数内容偏浅，容易泛化；
- AI 不理解当前产品已有功能、页面结构、设计规范、权限体系和业务边界；
- AI 生成原型容易自由发挥，产物看起来完整，但不符合真实产品体系；
- 已有产品能力没有被结构化沉淀，导致后续新需求难以复用上下文。

因此，OpsinTech 如果要做 AI 产品能力，不应直接做一个“PRD 生成器”或“原型生成器”，而应该先建立：

> **产品资产库 + 设计规范库 + 业务规则库 + AI 受约束生成流程。**

也就是说，让 AI 先理解当前产品，再辅助完成需求分析、方案设计、PRD 编写和页面结构生成。

---

## 2. 核心判断

### 2.1 通用 AI 产品工具为什么容易变浅

通用 AI 产品工具通常基于以下流程：

```text
用户输入想法
  ↓
AI 生成需求分析 / PRD / 原型
  ↓
用户复制或下载
```

这个流程的问题是：

- AI 不知道已有产品长什么样；
- AI 不知道哪些功能已经存在；
- AI 不知道已有页面入口和布局；
- AI 不知道当前产品的组件体系；
- AI 不知道权限、租户、文件、Agent、终端等核心业务对象；
- AI 不知道历史版本决策；
- AI 不知道哪些方案符合当前产品方向，哪些是不该做的；
- AI 会补充大量“看起来合理但实际不落地”的内容。

所以，真正的问题不是 AI 不会写文档，而是：

> **AI 没有被当前产品资产约束。**

### 2.2 OpsinTech 的机会点

OpsinTech 已经有正在运行的产品，包括：

- Web 工作台；
- 终端 Agent；
- Agent 对话；
- 命令执行与日志输出；
- 用户、租户、权限；
- 后续文件空间；
- 产品规划目录；
- 版本文档；
- 后端 Agent Runtime 雏形。

这些现有产品资产可以成为 AI 产品规划的基础上下文。

因此，OpsinTech 的 AI 产品能力不应定位为：

```text
通用 AI 产品经理
```

而应定位为：

```text
基于已有产品资产的产品规划智能工作台
```

核心差异是：

```text
不是 AI 自由创作，
而是 AI 在已有产品体系内推演、补充和生成。
```

## 3. 产品定位

### 3.1 产品名称建议

可选名称：

- 产品资产智能工作台；
- Product Intelligence Workspace；
- Product Asset Workspace；
- 产品设计知识库；
- 受约束 AI 产品规划系统。

当前建议使用：

> **产品资产智能工作台**

### 3.2 一句话定位

> 基于已有产品资产、设计规范、业务对象和历史版本，辅助团队进行需求分析、PRD 编写、页面结构设计和产品方案迭代。

### 3.3 不做什么

第一阶段不做：

- 通用 PRD 生成器；
- 通用竞品分析工具；
- 通用 Axure / Figma 替代品；
- 自由 HTML 原型生成器；
- 从 0 到 1 的独立产品设计工具；
- 完整项目管理平台；
- 完整研发 / 测试 / 运维全流程平台。

### 3.4 要做什么

第一阶段重点做：

- 沉淀当前产品资产；
- 建立产品地图；
- 建立页面字典；
- 建立组件与交互规范；
- 建立业务对象模型；
- 建立权限与边界规则；
- 建立 PRD 方法论模板；
- 让 AI 基于这些资产做需求澄清和 PRD 生成；
- 让 AI 生成受约束的页面结构，而不是自由发挥的原型。

## 5. 产品资产库设计

产品资产库是整个系统的核心。

它用于回答：

```text
当前产品是什么？
已经有哪些功能？
已有页面怎么组织？
已有组件怎么使用？
有哪些业务对象？
有哪些权限规则？
过去为什么这样设计？
新需求应该放在哪里？
哪些内容不能随便改？
```

### 5.1 产品地图 Product Map

产品地图用于描述当前产品的模块结构。

示例：

```text
OpsinTech
├── 工作台
├── 终端 Agent
├── Agent 会话
├── 文件空间
├── 产品规划
├── 用户与租户
├── 系统设置
└── 后续扩展模块
```

每个模块应包含：

| 字段 | 说明 |
| --- | --- |
| 模块名称 | 例如“终端 Agent” |
| 模块目标 | 该模块解决什么问题 |
| 目标用户 | 谁会使用该模块 |
| 核心功能 | 当前已有能力 |
| 页面列表 | 对应哪些页面 |
| 业务对象 | 涉及哪些数据模型 |
| 权限规则 | 谁能访问、谁能管理 |
| 当前状态 | 已上线 / 开发中 / 规划中 |
| 相关文档 | 关联 PRD、版本规划、技术方案 |

---

### 5.2 页面字典 Screen Dictionary

页面字典用于结构化描述已有页面。

不是简单截图，而是描述页面结构、区域、组件和交互。

示例：

```yaml
screen: 终端 Agent 页面
route: /workspace/terminal
purpose: 让用户通过 AI Agent 执行终端任务、查看命令输出并进行确认
regions:
  - name: 会话区
    purpose: 展示历史会话和当前任务
  - name: Agent 对话区
    purpose: 用户输入任务，Agent 返回计划和结果
  - name: 命令执行区
    purpose: 展示命令、状态、输出和审批动作
  - name: 日志区
    purpose: 展示流式输出和执行记录
interactions:
  - 用户输入任务
  - Agent 分析任务
  - 高风险命令等待用户确认
  - 命令执行结果流式返回
  - 会话记录保存
```

页面字典的价值是：

- AI 生成新功能时知道入口放在哪里；
- AI 不会重新设计一套完全不同的页面结构；
- AI 可以基于已有页面做增量设计；
- 后续可用于生成原型结构。

---

### 5.3 组件规范 Component Spec

组件规范用于约束 AI 生成页面时可以使用哪些组件。

示例组件：

```text
- 页面标题区 PageHeader
- 状态卡片 StatusCard
- 数据表格 DataTable
- 表单 Form
- 抽屉 Drawer
- 弹窗 Modal
- 标签 Badge
- Markdown 编辑器 MarkdownEditor
- 文件选择器 FilePicker
- Agent 对话面板 AgentChatPanel
- 命令确认卡片 CommandApprovalCard
- 版本列表 VersionList
- 上下文选择器 ContextPicker
```

每个组件需要描述：

| 字段 | 说明 |
| --- | --- |
| 组件名称 | 组件的标准名称 |
| 使用场景 | 什么时候使用 |
| 输入数据 | 需要哪些字段 |
| 交互行为 | 支持哪些操作 |
| 状态类型 | loading / empty / error / success |
| 禁止用法 | 不应该如何使用 |
| 示例页面 | 当前产品中哪里使用 |

组件规范的目标是：

> AI 不直接画 UI，而是用已有组件组装产品方案。

---

### 5.4 业务对象模型 Domain Model

业务对象模型用于让 AI 理解产品底层概念。

当前和未来可能涉及的对象：

```text
Tenant 租户
User 用户
Workspace 工作空间
TerminalSession 终端会话
AgentTask Agent 任务
CommandExecution 命令执行
FileAsset 文件资产
ProductSpace 产品空间
ProductArtifact 产品产物
ArtifactVersion 产物版本
Requirement 需求
DesignSpec 设计规范
PrototypeModel 原型结构模型
```

每个对象需要描述：

| 字段 | 说明 |
| --- | --- |
| 对象名称 | 业务对象名 |
| 定义 | 该对象代表什么 |
| 关键字段 | ID、名称、状态、归属等 |
| 生命周期 | 创建、更新、归档、删除 |
| 关联对象 | 与哪些对象有关 |
| 权限规则 | 谁能访问和操作 |
| 版本规则 | 是否需要版本 |
| 审计规则 | 是否需要记录操作日志 |

业务对象模型可以显著提升 PRD 质量。

没有对象模型，AI 生成的需求容易停留在表面；有对象模型后，AI 才能写出：

- 数据结构；
- 状态流转；
- 权限规则；
- 关联关系；
- 边界条件；
- 验收标准。

---

### 5.5 权限与边界规则 Permission Rules

权限规则是 OpsinTech 的重要产品约束。

当前建议采用三档可见性：

```text
private：仅自己可见
tenant：租户内可见
admin：租户管理员可管理
```

AI 生成任何新功能时，都必须考虑：

- 谁可以创建；
- 谁可以查看；
- 谁可以编辑；
- 谁可以删除；
- 谁可以发布为租户可见；
- 是否需要管理员审批；
- 是否需要操作审计；
- 是否影响租户配额；
- 是否涉及敏感信息。

权限规则应作为 AI Prompt 的强约束，而不是 PRD 后补内容。

### 5.6 设计语言 Design Language

设计语言不只是视觉风格，也包括交互模式。

需要沉淀：

- 页面布局模式；
- 导航结构；
- 表单组织方式；
- 空状态文案；
- 加载状态；
- 错误提示；
- 危险操作确认；
- Agent 对话风格；
- 命令审批风格；
- 文档编辑体验；
- 版本对比体验。

AI 生成页面结构时，必须遵守设计语言。

## 6. 方法论体系

产品资产智能工作台不能只依赖大模型自由发挥，需要内置方法论。

### 6.1 需求澄清方法

AI 在生成 PRD 前必须先澄清：

```text
1. 目标用户是谁？
2. 当前痛点是什么？
3. 使用场景是什么？
4. 用户现在如何解决？
5. 目标结果是什么？
6. 成功指标是什么？
7. 是否影响已有功能？
8. 是否涉及权限和租户？
9. 是否涉及文件和存储？
10. 是否涉及安全风险？
11. V0.1 做什么？
12. 明确不做什么？
```

### 6.2 PRD 结构模板

建议标准 PRD 包含：

```text
1. 背景
2. 问题定义
3. 目标用户
4. 使用场景
5. 产品目标
6. 非目标
7. 现有产品影响
8. 功能范围
9. 用户流程
10. 页面结构
11. 数据对象
12. 权限规则
13. 状态流转
14. 异常场景
15. 验收标准
16. 版本规划
17. 风险与待确认问题
```

### 6.3 页面设计方法

AI 生成页面方案时，应遵循：

```text
1. 先判断是否已有页面可以承载；
2. 优先增量改造已有页面；
3. 优先复用已有组件；
4. 必须说明入口位置；
5. 必须说明核心交互；
6. 必须说明状态和异常；
7. 必须说明权限差异；
8. 不直接生成最终 UI，而是生成结构化页面模型。
```

## 7. 原型生成策略

### 7.1 为什么不直接生成 HTML

直接生成 HTML 存在问题：

- 不可持续编辑；
- 难以和真实组件体系一致；
- 容易变成一次性 Demo；
- 难以表达权限、状态、异常和业务规则；
- 后续无法稳定转换为开发任务；
- AI 容易自由发挥视觉和交互。

因此，不建议把 HTML 作为源产物。

### 7.2 推荐的三层原型模型

```text
需求描述
  ↓
页面结构模型
  ↓
组件化渲染
  ↓
可视化预览
```

AI 主要生成第二层：页面结构模型。

示例：

```json
{
  "screen": "终端任务模板管理",
  "entry": "终端 Agent 页面右上角新增模板入口",
  "layout": "workspace_with_drawer",
  "regions": [
    {
      "name": "模板列表",
      "component": "DataTable",
      "fields": ["模板名称", "可见范围", "创建人", "更新时间", "操作"]
    },
    {
      "name": "模板编辑抽屉",
      "component": "DrawerForm",
      "fields": ["模板名称", "描述", "命令内容", "变量", "可见范围"]
    },
    {
      "name": "执行确认卡片",
      "component": "CommandApprovalCard",
      "behavior": "用户选择模板后生成命令，执行前必须确认"
    }
  ],
  "permissions": {
    "private": "仅创建人可见",
    "tenant": "租户成员可使用",
    "admin": "租户管理员可管理全部模板"
  }
}
```

这样后续可以：

- 渲染成低保真原型；
- 转成 PRD 页面说明；
- 转成开发任务；
- 和真实组件库对齐；
- 支持版本化修改。

## 8. AI 工作流设计

### 8.1 输入

用户输入可以是：

- 一句话想法；
- 一段讨论记录；
- 已有腾讯文档内容；
- 本地 Markdown；
- 版本规划；
- 用户反馈；
- 竞品截图或描述；
- 现有页面说明；
- 代码或接口约束。

### 8.2 上下文选择

AI 不应默认读取所有内容，建议用户显式选择上下文：

```text
- 选择产品模块；
- 选择相关页面；
- 选择已有 PRD；
- 选择版本规划；
- 选择设计规范；
- 选择相关文件；
- 选择历史对话。
```

### 8.3 处理流程

```text
用户输入需求
  ↓
AI 识别相关产品模块
  ↓
AI 检索产品资产库
  ↓
AI 提出澄清问题
  ↓
用户补充关键信息
  ↓
AI 生成结构化需求模型
  ↓
AI 生成 PRD 草稿
  ↓
AI 生成页面结构模型
  ↓
用户评审与修改
  ↓
保存为 Product Artifact
  ↓
生成版本记录
```

### 8.4 输出

输出不应只有 Markdown，还应包含结构化数据。

```text
Markdown：用于阅读、评审、编辑；
JSON：用于 AI 理解、版本 diff、页面结构渲染；
元数据：用于关联产品模块、页面、组件和版本。
```

## 12. 风险与约束

### 12.1 风险：产品资产维护成本高

如果产品资产库不能持续更新，就会失效。

缓解方式：

- V0.1 先手工维护；
- V0.4 开始半自动抽取路由、组件、模型；
- 每个需求变更后更新相关资产；
- 将资产更新纳入 PRD 流程。

### 12.2 风险：AI 仍然自由发挥

如果 Prompt 约束不够，AI 仍会生成脱离现实的方案。

缓解方式：

- 强制 AI 引用产品资产；
- 输出中标明依据；
- 不允许无依据扩展核心功能；
- 对未知内容标记为“待确认”；
- 页面方案必须使用已有组件或说明新增组件原因。

### 12.3 风险：原型系统复杂度过高

完整原型编辑器复杂度很高。

缓解方式：

- 第一阶段只做页面结构模型；
- 不做高保真自由画布；
- 不直接生成 HTML 作为源产物；
- 先满足 PRD 和研发沟通。

### 12.4 风险：文档工具化而非系统化

如果只是多生成几份 Markdown，仍然会回到浅层 AI 工具。

缓解方式：

- Markdown 只是展示层；
- 必须保留结构化 JSON；
- 必须支持版本；
- 必须关联模块、页面、组件、对象；
- 必须支持后续继续迭代。

## 14. 总结

OpsinTech 做 AI 产品能力的机会，不在于做一个通用 AI 产品经理，也不在于让 AI 凭空生成 PRD 或原型。

真正的机会在于：

> **把当前运行中的产品沉淀为 AI 可理解的产品资产库，让 AI 在已有产品体系、设计规范、业务规则和历史决策约束下，辅助完成需求分析、PRD 编写和页面结构设计。**

这条路线的核心价值是：

```text
从“AI 自由生成”转向“AI 基于产品资产推演”；
从“一次性文档”转向“可持续演进的产品知识库”；
从“看起来完整”转向“真正贴合当前产品、可评审、可落地”。
```

如果这个体系能先服务好 OpsinTech 自己的产品规划，再考虑对外产品化，成功概率会比直接进入通用 AI 产品经理市场更高。
