# 用户与租户文件空间版本规划

> 目标：为 OpsinTech 平台建设一套通用的 File Center / Storage Center，支持租户配额、用户私有文件、租户公共文件、业务附件和 AI 生成产物，并为后续接入 MinIO、文档解析、知识库检索打好基础。

## 1. 背景

当前平台还没有给租户和用户建立统一的存储空间。现阶段更多依赖本地目录、浏览器 localStorage 或业务模块自身的数据结构保存状态。

随着后续 Web 端能力增强，尤其是 AI 产品经理工作台、智能终端、智能告警、资产管理、知识库等模块发展，平台会持续产生和使用大量文件：

- 用户上传的需求资料、竞品资料、会议纪要、图片、PDF、Office 文档；
- AI 生成的 PRD、竞品分析、原型说明、报告、导出文件；
- 租户管理员上传的团队公共资料；
- 业务模块中的附件，例如告警附件、资产附件、工单附件、项目附件；
- 后续知识库解析后的文本、切片、索引与引用记录。

因此需要把“文件空间”做成平台通用基础能力，而不是 AI 产品工作台的私有功能。

## 2. 设计原则

### 2.1 通用基础设施优先

文件中心应服务全平台：AI 工作台、告警、资产、工单、知识库、项目管理等模块都可以复用。

### 2.2 租户隔离优先

平台管理员给租户分配总存储空间，所有文件都必须带 `tenant_id`，遵循现有后端模型的多租户隔离方式。

### 2.3 用户隐私默认安全

用户上传或 AI 自动生成的文件默认属于用户私有文件，只有本人可见。用户确认后，才可以发布为租户可见。

### 2.4 存储实现与业务权限解耦

业务权限不依赖真实文件路径或 MinIO bucket policy，而是由数据库元数据控制。

底层存储只负责保存二进制对象：

- V0.1：local storage driver；
- V0.2：MinIO storage driver；
- 长期：支持 driver 配置切换。

### 2.5 先可靠，再智能

第一版优先保证上传、下载、权限、配额、审计、解析可靠。AI 检索和向量化分阶段演进，避免过早承诺准确性和性能。

## 3. 目标用户与使用场景

### 3.1 平台管理员

- 创建租户；
- 给租户分配总存储空间；
- 查看租户用量；
- 调整租户配额；
- 管理存储后端配置。

### 3.2 租户管理员

- 查看本租户总用量；
- 上传租户公共资料；
- 管理租户内公共文件；
- 后续为租户内用户分配个人配额；
- 控制哪些文件或文件夹可进入知识库。

### 3.3 普通用户

- 上传个人文件；
- 查看自己的私有文件；
- 查看租户公共文件；
- 使用文件作为 AI 对话上下文；
- 保存 AI 生成的文件；
- 将确认后的 AI 产物发布到租户空间。

### 3.4 AI Agent / 智能工作台

- 生成 PRD、竞品分析、原型说明等文件；
- 默认把产物保存为用户私有文件；
- 在用户显式授权后读取文件内容；
- 后续在知识库范围内执行权限感知检索。

## 4. 核心产品模型

## 4.1 空间归属

采用“租户配额 + 用户权限”的混合模型。

```text
平台管理员
  └── 给租户分配总存储空间
        ├── 用户私有文件：仅 owner_user_id 可见
        ├── 租户公共文件：租户内用户可见
        ├── 业务附件：关联具体业务对象
        └── AI 产物：默认归属于创建用户
```

## 4.2 文件可见性

第一版采用三档权限语义，但底层建议用两个字段表达：

```text
visibility = private | tenant
owner_user_id = 上传者 / 创建者
created_by_role = user | tenant_admin | platform_admin
```

权限语义：

| 类型 | 建议表达 | 可见范围 | 说明 |
|---|---|---|---|
| 用户私有文件 | `visibility=private` | 仅创建用户 | 用户上传、AI 草稿默认值 |
| 租户公共文件 | `visibility=tenant` | 租户内用户 | 租户管理员或用户发布的公共资料 |
| 管理员文件 | `visibility=tenant` + `created_by_role=tenant_admin` | 租户内用户可读，管理员可管理 | 不单独增加复杂 ACL |

暂不做：

- 指定用户分享；
- 外链分享；
- 部门/项目组权限；
- 复杂 RBAC / ABAC。

这些能力后续可扩展。

## 4.3 虚拟目录树

推荐采用虚拟目录树，而不是直接映射真实服务器目录。

用户看到：

```text
/我的文件/需求文档/AI竞品分析PRD.md
/租户空间/公共资料/竞品资料.pdf
```

数据库记录：

```text
file_id
tenant_id
owner_user_id
visibility
parent_folder_id
display_name
object_key
storage_backend
mime_type
size_bytes
checksum
created_at
updated_at
```

底层存储：

```text
local: data/storage/tenant_xx/objects/ab/cd/hash
minio: bucket/tenant_xx/objects/ab/cd/hash
```

优势：

- 列表性能依赖数据库索引，不扫描磁盘或 MinIO prefix；
- 权限由数据库统一控制；
- 后续 local / MinIO 可无缝切换；
- 可支持回收站、重命名、移动、标签、业务关联；
- 避免用户通过路径猜测访问文件。

## 5. 存储后端策略

## 5.1 StorageService 抽象

第一版就引入统一接口：

```text
StorageService
  ├── put_object(stream, metadata) -> object_key
  ├── get_object(object_key) -> stream
  ├── delete_object(object_key)
  ├── exists(object_key)
  ├── get_size(object_key)
  └── generate_download_url(object_key, expires_in)
```

业务代码只依赖 `StorageService`，不直接操作本地路径或 MinIO SDK。

## 5.2 Driver 演进

| 版本 | Driver | 说明 |
|---|---|---|
| V0.1 | LocalStorageDriver | 开发和早期部署使用，简单可靠 |
| V0.2 | MinIOStorageDriver | 面向生产部署，对象存储能力更完整 |
| V0.3 | Hybrid / Migration Tool | 支持从 local 迁移到 MinIO |

配置示例：

```yaml
storage:
  backend: local
  local:
    root_path: ./data/storage
  minio:
    endpoint: http://minio:9000
    bucket: opsintech-files
    access_key: ${MINIO_ACCESS_KEY}
    secret_key: ${MINIO_SECRET_KEY}
```

## 5.3 上传下载链路

第一版采用后端中转上传下载。

```text
前端上传文件
  ↓
FastAPI 接收
  ↓
校验登录态 / tenant_id / 权限 / 配额
  ↓
写入 StorageService
  ↓
创建 file_record / file_version
  ↓
提交事务
```

优点：

- 权限、配额、审计统一；
- 失败回滚简单；
- 不暴露底层存储；
- 适合 PRD、PDF、Office、图片等常见文档。

建议第一版限制：

```text
单文件 <= 50MB
```

后续大文件再支持：

```text
> 50MB：MinIO 预签名 multipart upload
```

## 6. 配额模型

长期最佳方案是“租户总配额 + 用户个人配额”。

建议分阶段：

| 阶段 | 能力 |
|---|---|
| V0.1 | 强制租户总配额，预留用户配额字段 |
| V0.2 | 开启用户个人配额管理 |
| V0.3 | 支持配额预警、超限策略、统计报表 |

### 6.1 租户配额

平台管理员为每个租户设置：

```text
tenant_storage_quota_bytes
tenant_storage_used_bytes
```

上传前检查：

```text
used_bytes + incoming_file_size <= quota_bytes
```

### 6.2 用户配额

租户管理员后续可为用户设置：

```text
user_storage_quota_bytes
user_storage_used_bytes
```

第一版可先记录用户用量，但不强制。

## 7. 文件版本管理

推荐：重要文档支持版本，普通附件不做复杂版本。

### 7.1 适合版本管理的文件

- PRD；
- Markdown 文档；
- 竞品分析报告；
- 原型说明；
- 方案文档；
- AI 生成并可继续编辑的文本文件。

### 7.2 普通附件

以下文件默认只作为普通文件处理：

- 图片；
- 压缩包；
- 日志包；
- 原始导入资料；
- 视频 / 音频。

### 7.3 版本记录字段

```text
file_versions
  id
  file_id
  tenant_id
  version_no
  object_key
  size_bytes
  checksum
  source_type: upload | ai_generated | manual_edit | restore
  created_by
  created_at
  change_summary
```

能力范围：

- 查看版本列表；
- 标记当前版本；
- 恢复历史版本；
- 记录版本来源；
- AI 重写生成新版本。

## 8. 预览与解析

为了让文件中心不只是网盘，第一版建议支持“基础预览 + 核心文档解析”。

### 8.1 预览范围

| 类型 | 第一版能力 |
|---|---|
| Markdown / TXT | 直接预览 |
| PDF | 浏览器内嵌预览 |
| 图片 | 直接预览 |
| DOCX | 可下载，后续增强预览 |
| Excel / PPT | 可下载，后续增强 |

### 8.2 解析范围

| 类型 | 第一版能力 |
|---|---|
| Markdown / TXT | 直接抽取文本 |
| PDF | 抽取正文文本，先不做复杂版式还原 |
| DOCX | 抽取正文文本 |
| 图片 OCR | 暂缓 |
| Excel 深度解析 | 暂缓 |
| PPT 结构化解析 | 暂缓 |
| 视频 / 音频解析 | 暂缓 |

### 8.3 异步解析流程

```text
文件上传成功
  ↓
创建 file_record
  ↓
创建 parse_job
  ↓
后台 worker 解析
  ↓
写入 file_text_contents / document_chunks
  ↓
后续可进入关键词索引或向量索引
```

不要在上传接口中同步解析大文件，避免接口超时和用户体验不稳定。

## 9. AI 使用文件策略

### 9.1 默认策略

第一版 AI 使用文件时，默认采用“用户显式选择文件”。

```text
用户在 AI 输入框点击“添加文件”
  ↓
选择自己有权限的文件
  ↓
AI 只读取这些文件的解析文本
  ↓
生成回答或新文件
```

优势：

- 权限最安全；
- 性能可控；
- 结果可解释；
- 用户知道 AI 看了哪些资料。

### 9.2 预留策略

后续支持：

```text
选择文件夹 / 知识库范围
  ↓
AI 在该范围内检索
```

再后续支持：

```text
允许 AI 自动检索相关资料
  ↓
仅在用户有权限的范围内检索
```

界面必须明确提示：

```text
AI 将仅在你有权限的这些文件/文件夹中检索。
```

### 9.3 AI 产物保存策略

AI 生成的 PRD、竞品报告、原型说明等文件，默认保存为用户私有文件：

```text
/我的文件/AI产物/PRD草稿/xxx.md
```

用户确认后，可以发布到租户空间：

```text
/租户空间/项目资料/xxx.md
```

## 10. 检索与知识库演进

针对性能和准确性，建议不要第一版默认全量向量化。

### 10.1 分阶段策略

| 阶段 | 能力 |
|---|---|
| V0.1 | 文本抽取 + 权限过滤 + 显式文件使用 |
| V0.2 | 关键词检索 / BM25，支持在文件夹或知识库范围搜索 |
| V0.3 | 可选向量化，对指定知识库开启 |
| V0.4 | 混合检索：关键词 + 向量 + 重排 |
| V0.5 | 检索审计、引用溯源、质量评估 |

### 10.2 企业级检索原则

权限过滤必须发生在检索阶段，而不是先检索再过滤。

错误方式：

```text
先从所有文档 top-k 检索
  ↓
再过滤用户无权限文档
```

问题：

- 可能泄露私有文档存在；
- 无权限内容会挤占 top-k；
- 结果不稳定；
- 合规风险高。

正确方式：

```text
构造查询时携带 tenant_id / owner_user_id / visibility / folder_scope
  ↓
索引层只在有权限范围内召回
  ↓
再排序和生成答案
```

### 10.3 向量化策略

不建议第一版所有文件自动向量化。

推荐：

```text
默认不向量化
  ↓
租户管理员或用户选择文件夹进入知识库
  ↓
后台生成 chunks + embeddings
  ↓
查询时按权限和范围过滤
```

原因：

- 控制 embedding 成本；
- 控制索引规模；
- 降低错误召回；
- 避免私有草稿被意外进入知识库；
- 便于用户理解 AI 的资料来源。

## 14. 版本路线图

目标：建立通用文件空间，支持上传、下载、权限、租户配额、基础预览与解析。

范围：

- StorageService 抽象；
- LocalStorageDriver；
- tenant_id 多租户隔离；
- 用户私有文件；

- 用户用量记录；
- 后端中转上传下载；

- MinIO；

- 音视频解析。

## 15. 风险与注意事项

### 15.1 权限风险

文件权限必须在所有入口统一校验：

- 列表；
- 详情；
- 下载；
- 预览；
- 解析文本读取；
- AI 检索；
- 版本查看；
- 删除 / 恢复。

不能只在前端隐藏按钮。

### 15.2 配额一致性

上传失败、删除文件、恢复文件、版本覆盖都涉及 used_bytes 变化。需要事务或补偿机制保证用量准确。

### 15.3 本地存储迁移

V0.1 使用 local driver 时也必须保存 `storage_backend` 和 `object_key`，不能直接在业务表里写真实路径，否则后续迁移 MinIO 成本高。

### 15.4 解析任务稳定性

解析任务必须异步，并记录失败原因。不能因为解析失败影响文件上传成功。

### 15.5 向量化成本与准确性

向量化不是越早越好。应该先明确：

- 哪些文件进入知识库；
- 哪些用户有权检索；
- 如何展示引用来源；
- 如何评估召回质量；
- 如何处理权限变更后的索引更新。
