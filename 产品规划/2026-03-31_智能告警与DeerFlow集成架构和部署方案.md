# 2026-03-31 智能告警与 DeerFlow 集成架构和部署方案

## 1. 文档目标

本文档回答两个最关键的工程问题：

1. 智能告警如何与当前 DeerFlow 平台集成。
2. 智能告警第一阶段应该如何部署。

---

## 2. 总体原则

第一阶段不建议把智能告警拆成独立大系统，而建议：

**先作为 DeerFlow 的新业务模块集成到现有平台内部。**

原因：

1. 当前平台已经有用户、租户、权限、审计和工作区底座
2. 智能告警第一阶段更需要验证产品闭环，而不是验证微服务架构
3. 智能告警天然需要和工作台、工作流、终端打通
4. 一开始拆太细会显著增加研发和部署复杂度

---

## 3. 集成架构

### 3.1 逻辑关系

```text
外部告警源
    ↓
告警接入层
    ↓
标准化与聚合层
    ↓
Incident 事件层
    ↓
智能工作台 / 工作流 / 智能终端
```

### 3.2 DeerFlow 内部模块关系

```text
Frontend Workspace
 ├─ 智能告警
 ├─ 工作台
 ├─ 工作流
 └─ 智能终端

Gateway / API
 ├─ Auth / RBAC / Tenant Context
 ├─ Alert Ingest API
 ├─ Incident API
 ├─ Workflow API
 └─ Terminal API

Core Services
 ├─ Alert Normalizer
 ├─ Incident Engine
 ├─ AI Triage
 ├─ Workspace Binder
 └─ Audit Service

Storage
 ├─ Auth DB
 ├─ Tenant DB / Incident Tables
 ├─ Thread / Artifact / Memory Storage
 └─ Audit Logs
```

## 4. 与当前平台的集成点

### 4.1 用户与租户体系

直接复用现有：

1. Better Auth
2. `user_id`
3. `tenant context`
4. 管理员与普通用户角色

智能告警模块不应自建独立账号体系。

### 4.2 工作区入口

前端在当前 workspace 中新增：

1. 智能告警列表页
2. 事件详情页
3. 从 incident 进入工作台的按钮

### 4.3 线程体系

建议 incident 和 DeerFlow thread 建立绑定关系：

1. 一个 incident 可绑定一个处置 thread
2. thread 中保留 incident context
3. 后续 AI 对话围绕该 incident 展开

### 4.4 工作流体系

incident 需要支持：

1. 触发工作流
2. 查看工作流结果
3. 把执行结果回流到 incident 时间线

### 4.5 智能终端体系

incident 打开终端时，需要把以下内容带入：

1. 服务名
2. 环境
3. 严重级别
4. 相关上下文摘要
5. 审批/权限信息

## 5. 第一阶段部署方案

### 5.1 推荐方案：单体内嵌部署

建议第一阶段继续沿用 DeerFlow 当前部署方式：

1. 前端与当前平台一起部署
2. 后端 gateway 增加智能告警 router
3. 数据库增加 incident 相关表
4. AI 能力继续复用当前模型配置

### 5.2 推荐部署结构

```text
DeerFlow
 ├─ frontend
 ├─ backend
 │   ├─ auth / gateway
 │   ├─ incidents
 │   ├─ alerts
 │   ├─ workflows
 │   └─ terminal
 └─ database
```

### 5.3 第一阶段为什么不拆服务

因为现在真正要验证的是：

1. 用户是否需要 incident 工作台
2. 智能摘要是否真的提升处理效率
3. incident 能否顺利流入工作流和终端

这些问题和微服务拆分没有直接关系。

---

## 6. 第二阶段可演进部署方案

当智能告警接入量、处理量和异步任务明显上升后，可演进为：

### 6.1 可拆出的服务

1. `alert-ingest-service`
2. `incident-engine-service`
3. `ai-triage-worker`

### 6.2 何时拆分

以下情况满足其一时再考虑：

1. ingest 吞吐明显高
2. incident 聚合和 AI 分析变得异步且耗时
3. 需要独立扩缩容
4. 需要专门队列处理

### 6.3 推荐演进方式

先单体模块化，再服务化。

不要从第一天就做成多服务。

## 8. 事件到工作台的绑定方案

### 8.1 推荐做法

新增一个 incident 与 thread 的绑定关系，例如：

1. 一个 incident 首次进入工作台时，创建 thread
2. 后续再次打开，直接回到原 thread

### 8.2 绑定时传递的信息

建议写入 thread metadata：

1. `incident_id`
2. `incident_key`
3. `service`
4. `environment`
5. `severity`
6. `ai_summary`

### 8.3 好处

1. 用户对话和 incident 形成闭环
2. 后续工作流和终端更容易接入
3. 历史处置记录更容易沉淀

## 9. 第一阶段部署与运维建议

### 9.1 本地开发

建议：

1. 通过 mock webhook 方式压入告警
2. 用本地 alertmanager 模拟真实场景
3. 前后端都跑在当前 DeerFlow 开发环境内

### 9.2 测试环境

建议：

1. 先接一个真实但低风险的告警源
2. 观察 incident 聚合效果
3. 观察 AI 摘要质量
4. 观察工作台跳转链路

### 9.3 生产环境

建议：

1. 先以旁路方式接入
2. 先只看 incident 和 AI 摘要
3. workflow / terminal 的自动执行后开

---

## 10. 业界设计借鉴与本平台取舍

### 借鉴点

1. Grafana IRM 的 alert group / incident 分层
2. PagerDuty 的 incident workflow 触发和动作串联
3. Splunk 的嵌入式 AI 助手
4. Dynatrace 的稳定问题对象页
5. BigPanda 的上下文丰富 incident

### DeerFlow 的取舍

1. 第一阶段不做完整 on-call
2. 第一阶段不做复杂拓扑根因
3. 第一阶段重点做 incident 事件引擎和工作台联动
4. 执行面后接 workflow 和 terminal，形成差异化

---

## 11. 最终建议

智能告警最适合 DeerFlow 的集成方式，不是外挂一个告警系统，而是：

**把它做成 DeerFlow 平台内部的 incident 业务域，并以此连接工作台、工作流和终端。**

第一阶段的最佳部署方式，就是继续与当前 DeerFlow 一体化部署。
