## 三个能力的实现方式（和传统平台的区别）

### 1. 认领 → 工单

**传统告警平台**：认领 = 内部状态流转（ack），工单是另一个系统。

**你的做法**：认领动作触发数字人（或规则）**通过 bound tool 向外发工单**。

```
用户在告警详情页点"认领"
  → incident.accepted_by = user_id
  → 触发 Agent action: "创建工单"
  → Agent 调用 bound http tool → Jira/飞书/企微 API 创建工单
  → incident.ticket_id = 外部工单 ID
  → incident.ticket_url = 外部工单链接
```

incident 表增加：

```
├── accepted_by       # 认领人
├── accepted_at       # 认领时间
├── ticket_id         # 外部工单 ID
└── ticket_url        # 外部工单链接
```

这不是 DeerFlow 自己做工单系统，而是**把工单作为外部动作的产物**。数字人的 bound tool 负责对接外部工单 API。

### 2. 告警链路升级

**传统告警平台**：复杂的多级 escalation policy，按时间梯度通知不同人。

**你的做法**：升级规则 + 数字人通知。

```
alert_rules 表（已有设计）扩展：
├── rule_type: "escalation"  # 新增类型
├── condition_json:
│   {
│     "severity": "critical",
│     "unacknowledged_minutes": 30
│   }
└── action_json:
    {
      "action": "notify",
      "targets": ["tenant_admin", "wechat_group_xxx"],
      "message_template": "..."
    }

或者更灵活的——升级触发数字人：
    {
      "action": "trigger_agent",
      "agent_name": "ops-escalation-agent"
    }
```

关键区别：升级不是固定通知链，而是**可以触发数字人做智能判断**——比如数字人先看一眼告警是否已经自愈，再决定要不要升级。

### 3. 静默规则

**传统告警平台**：按标签/服务/时间窗口静默。

**你的做法**：基本一致，但属于 `alert_rules` 的一种 rule_type。

```
alert_rules:
├── rule_type: "suppress"
├── condition_json:
│   {
│     "service": "batch-job-*",
│     "time_window": "02:00-06:00",
│     "labels": {"env": "staging"}
│   }
└── action_json:
    {
      "action": "suppress",
      "duration_minutes": 240,
      "note": "凌晨批处理已知抖动"
    }
```

静默规则在 ingest → signal → incident 链路的 dedup/aggregation 阶段生效。被静默的 raw_alert 仍然入库（可审计），但不生成 incident。

## 修正后的 incident 状态机

之前我说太简单的 `firing/resolved` 不够，你说得对。保持精简但不残缺：

```
状态流转：
  firing ──→ acknowledged ──→ resolved ──→ closed
    │            │                │
    │            │                │
    └── suppressed (被静默规则命中)
```

| `resolved` | 已解决，等待关闭 |
| `closed` | 已关闭 |
| `suppressed` | 被静默规则拦截 |

## 为什么普通用户必须能做"认领"

这点很重要。在真实的运维场景里：

```
认领告警 ≠ 管理告警
认领告警 = "这条我在看了，其他人别重复处理"
```

这是协作行为，不是管理行为。普通用户每天都做这件事——看到一个告警，点认领，开始排查。如果不让普通用户认领，那这个系统就没有使用者了，只有管理者。

```
  │   └── 点进 critical 那条

  │   ├── 关联 signal 列表（3 条原始告警聚合而成）
```

## 普通用户不能做的事（关键安全边界）

```yaml
普通用户不能:
  - 创建/编辑/删除告警源:
      原因: 告警源配置影响整个租户的数据接入，属于基础设施
  - 创建/编辑/删除静默规则:
      原因: 静默规则会屏蔽告警，错误配置可能导致关键告警丢失
  - 创建/编辑/删除升级规则:
      原因: 升级链路涉及通知策略，属于治理范畴
  - 删除/归档 incident:
      原因: 告警记录属于租户审计资产，不应被普通成员删除
  - 修改绑定的数字人 Agent:
      原因: Agent 决定了分析逻辑和可用工具，属于治理配置
```

## 静默规则为什么还值得做

静默规则不是"告警操作"，它是**分析策略**——决定什么告警值得进入分析管线。

```
静默规则 = "这个告警不值得数字人花 token 去分析"
```

属于配置，不属于操作。放在 DeerFlow 里合理。

## 升级链路怎么处理

DeerFlow 不做传统升级链路（按时间梯度通知）。但数字人可以**智能判断升级必要性**：

```
数字人分析后判断：
  "这个告警影响核心支付链路，建议立即升级"

用户在分析结果里看到这个建议
  → 自己在工单系统里加急/升级
```

或者更自动的——数字人分析完，如果判断需要升级，直接用 bound tool 在外部系统里标记优先级。**判断在 DeerFlow，执行在外部**。
