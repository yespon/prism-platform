# 2026-03-31 智能告警后端表结构与 API 草案

## 1. 文档目标

本文档用于给智能告警功能提供可直接进入开发的后端草案，包括：

1. 数据表结构建议
2. 核心状态机
3. API 清单
4. 与当前 DeerFlow 用户体系和工作台的集成约束

---

## 2. 设计原则

### 2.1 对象分层

必须区分：

1. `raw_alert`
2. `signal`
3. `incident`

### 2.2 多租户优先

所有核心表都必须显式带：

1. `tenant_id`
2. 必要时 `owner_user_id`
3. 审计相关字段

### 2.3 可接工作台

`incident` 必须成为 DeerFlow 工作台的上游对象。

### 2.4 可接工作流与终端

`incident` 必须支持：

1. workflow trigger
2. terminal context injection

---

## 4. 状态机草案

### 4.1 Signal 状态

建议仅保留：

1. `firing`
2. `resolved`

### 4.2 Incident 状态

建议保留：

1. `new`
2. `triaging`
3. `in_progress`
4. `mitigated`
5. `resolved`
6. `closed`
7. `suppressed`

### 4.3 状态流转原则

1. `new -> triaging`
2. `triaging -> in_progress`
3. `in_progress -> mitigated`
4. `mitigated -> resolved`
5. `resolved -> closed`
6. 任意合规状态可进入 `suppressed`

## 5. API 草案

### 5.1 接入类 API

#### `POST /api/alerts/ingest/webhook`

作用：

1. 接收通用 webhook 告警

请求体：

1. 任意 JSON

响应：

1. `202 Accepted`
2. 返回接收结果和 ingest id

#### `POST /api/alerts/ingest/alertmanager`

作用：

1. 接收 Alertmanager webhook

请求体：

1. Alertmanager 标准 payload

响应：

1. `202 Accepted`

### 5.2 查询类 API

#### `GET /api/incidents`

作用：

1. 获取事件列表

支持参数：

1. `status`
2. `severity`
3. `service`
4. `environment`
5. `owner_user_id`
6. `keyword`
7. `page`
8. `page_size`

#### `GET /api/incidents/{incident_id}`

作用：

1. 获取事件详情

返回：

1. incident 基本信息
2. AI 摘要
3. 影响说明
4. 建议动作

#### `GET /api/incidents/{incident_id}/signals`

作用：

1. 获取事件关联信号

#### `GET /api/incidents/{incident_id}/context`

作用：

1. 获取上下文快照

#### `GET /api/incidents/{incident_id}/actions`

作用：

1. 获取事件操作时间线

### 5.3 操作类 API

#### `POST /api/incidents/{incident_id}/assign`

作用：

1. 指派负责人

#### `POST /api/incidents/{incident_id}/status`

作用：

1. 更新事件状态

#### `POST /api/incidents/{incident_id}/suppress`

作用：

1. 静默/抑制事件

#### `POST /api/incidents/{incident_id}/re-analyze`

作用：

1. 重新触发 AI 分析

#### `POST /api/incidents/{incident_id}/open-workspace`

作用：

1. 进入 DeerFlow 工作台
2. 生成或绑定一个工作线程

返回：

1. `thread_id`
2. `workspace_url`

#### `POST /api/incidents/{incident_id}/trigger-workflow`

作用：

1. 触发工作流

#### `POST /api/incidents/{incident_id}/open-terminal`

作用：

1. 打开 incident 绑定终端上下文

### 5.4 管理类 API

#### `GET /api/alert-sources`
#### `POST /api/alert-sources`
#### `PATCH /api/alert-sources/{source_id}`
#### `GET /api/alert-rules`
#### `POST /api/alert-rules`
#### `PATCH /api/alert-rules/{rule_id}`

## 6. 与当前平台的集成约束

### 6.1 认证与权限

全部复用 DeerFlow 现有：

1. 登录态
2. 租户上下文
3. `user_id`
4. 管理员和普通用户角色

### 6.2 工作台联动

`incident` 进入工作台时，建议创建或绑定：

1. `thread_id`
2. `incident_id`
3. `tenant_id`
4. `owner_user_id`

### 6.3 workflow 联动

工作流触发时，建议把以下字段注入执行上下文：

1. `incident_id`
2. `service`
3. `environment`
4. `severity`
5. `context_snapshot`

### 6.4 terminal 联动

终端执行前必须校验：

1. 用户权限
2. 事件状态
3. 执行风险级别
4. 审批状态

## 8. MVP 最小实现建议

### 第一批最小后端能力

1. ingest webhook
2. raw_alert 入库
3. signal 标准化
4. incident 创建
5. incident 列表查询
6. incident 详情查询

### 第一批最小前端能力

1. 智能告警导航入口
2. incident 列表页
3. incident 详情页
4. 进入工作台按钮

### 第一批 AI 能力

1. AI 摘要
2. 影响说明
3. 建议下一步动作

---

## 9. 最终建议

第一阶段最重要的，不是把规则系统做复杂，而是：

1. 先把 `incident` 对象做对
2. 先把和 DeerFlow 工作台的接口打通
3. 先把 AI 用在事件理解，而不是高风险自动执行

只要这三点做对，后面的工作流和终端都能自然接进来。
