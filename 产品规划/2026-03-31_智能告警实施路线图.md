# 2026-03-31 智能告警实施路线图

## 1. 文档目标

本文档用于回答三个落地问题：

1. 智能告警功能应该如何启动。
2. 智能告警如何与当前 DeerFlow 平台集成。
3. 智能告警如何分阶段实现，而不是一开始做成大而重的平台。

---

## 2. 总体判断

当前 DeerFlow 已经具备以下底座能力：

1. 用户体系
2. 多租户隔离
3. 权限与管理员治理
4. 审计与资源隔离基础
5. 工作区与线程体系
6. Agent / Memory / Sandbox 基础设施

因此，智能告警不需要作为一个完全独立的新产品启动，而应作为 DeerFlow 的一个新业务域集成进来。

第一阶段不建议做成：

1. 完整告警平台
2. 完整 on-call 平台
3. 完整 AIOps 平台
4. 复杂 ML 根因平台

第一阶段建议只做：

**外部告警接入 -> 事件生成 -> AI 摘要 -> 进入 DeerFlow 工作台**

只要这条链路跑通，产品就已经开始成立。

---

## 3. 借鉴业界主流产品的设计原则

### 3.1 值得借鉴的设计

#### 借鉴一：Grafana IRM 的 Alert Group / Incident 分层

Grafana IRM 明确区分了：

1. Alert Group
2. Incident

Alert Group 用于承接技术信号，Incident 用于承接协同与响应。

这对 DeerFlow 的启发是：

1. 不要把所有对象都叫 alert
2. DeerFlow 应显式区分 `signal` 与 `incident`
3. 用户真正处理的核心对象应是 `incident`

#### 借鉴二：PagerDuty 的 Incident Workflow

PagerDuty 强调：

1. Trigger
2. Condition
3. Action
4. Sequential workflow

这对 DeerFlow 的启发是：

1. 告警系统不应止步于展示和状态流转
2. incident 必须天然可以进入 workflow
3. 后续工作流设计要从第一天就考虑 incident 触发

#### 借鉴三：Splunk AI Assistant 的“在现有工作流内对话”

Splunk 的 AI Assistant 并不是独立孤岛，而是嵌在现有可观测性工作流中，围绕 metrics、traces、logs、alerts 给出解释和探索能力。

这对 DeerFlow 的启发是：

1. AI 不要单独做一个聊天页
2. AI 应嵌在 incident 详情页和工作台内部
3. AI 的价值在于上下文解释，不只是文本生成

#### 借鉴四：Dynatrace 的 Root Cause + Context View

Dynatrace Problems 强调：

1. 一个稳定的 incident/problem 视图
2. 自动收集上下文
3. 在统一视图中呈现根因、影响和证据

这对 DeerFlow 的启发是：

1. incident 详情页必须是一个稳定对象页
2. 不能只是告警字段面板
3. 应自动呈现影响范围、时间线、关联信号、建议动作

#### 借鉴五：BigPanda 的事件关联与上下文聚合

BigPanda 强调：

1. 关联 alerts、changes、topology
2. 降噪
3. 形成 context-rich incidents

这对 DeerFlow 的启发是：

1. 智能告警必须服务于降噪和上下文构造
2. 不要把 AI 只用在摘要上
3. 上下文拼装本身就是产品价值

### 3.2 不建议照搬的设计

1. 不建议第一阶段就做完整 on-call 排班体系
2. 不建议第一阶段就做复杂可视化拖拽规则平台
3. 不建议第一阶段就做过重的依赖拓扑根因图
4. 不建议第一阶段让 AI 自动执行高风险动作

## 4. DeerFlow 的启动方式

### 4.1 正确的启动目标

第一阶段不是“做一个新的 alerta”，而是：

**把 DeerFlow 现有平台扩展出 incident 能力。**

### 4.2 第一阶段要解决的核心问题

1. 怎么把外部告警接进来
2. 怎么把外部告警转换成 DeerFlow 事件
3. 怎么让事件被 AI 解释
4. 怎么让事件进入 DeerFlow 工作台

### 4.3 第一阶段结束时应该看到的产品效果

1. 外部监控源可以向 DeerFlow 发告警
2. DeerFlow 可以生成 incident 列表
3. 用户点进事件能看到 AI 摘要和上下文
4. 用户可以进入工作台继续处置

---

## 5. 与当前平台的集成方案

### 5.1 业务域划分

建议在 DeerFlow 中新增“智能告警”业务域：

1. 后端新增 `alerts / incidents` 相关模块
2. 前端 workspace 新增“智能告警”入口
3. incident 作为后续工作流、终端、工作台的上游对象

### 5.2 推荐的集成关系

#### 用户体系

复用当前：

1. 登录
2. 多租户
3. 角色权限
4. 管理员治理

#### AI 能力

复用当前：

1. 模型配置
2. Agent 基础能力
3. 对话能力
4. Memory / Context 基础设施

#### 工作区体系

复用当前：

1. workspace 导航
2. 线程 / 任务上下文
3. 聊天与协作区域

#### 执行能力

后续复用：

1. workflow
2. terminal
3. audit

### 5.3 集成后的平台关系

建议关系如下：

1. 智能告警负责事件发现与生成
2. 智能工作台负责事件承接与协作
3. 智能工作流负责执行编排
4. 智能终端负责执行落地

## 6. 分阶段路线图

### 阶段 0：模型与集成设计

目标：先把底层对象模型和接口方向定住。

交付：

1. `raw_alert / signal / incident` 模型
2. 租户与权限设计
3. 核心 API 草案
4. 页面原型边界

状态标准：

1. 团队对对象模型达成一致
2. 前后端边界清楚

### 阶段 1：Incident MVP

目标：把“告警接入”升级成“事件入口”。

交付：

1. webhook 接入
2. Alertmanager 接入
3. raw alert 入库
4. signal 标准化
5. dedup / 基础聚合
6. incident 列表页
7. incident 详情页
8. AI 摘要
9. 进入工作台按钮

状态标准：

1. 可以接入真实告警
2. 用户可以看到 incident
3. 用户可以基于事件开始处理

### 阶段 2：Incident Workspace 联动

目标：让 incident 成为 DeerFlow 工作区入口。

交付：

1. incident 到工作台的上下文跳转
2. incident 时间线
3. 责任人指派
4. 历史同类事件
5. 最近变更/上下文聚合
6. 基础审计

状态标准：

1. 告警处理不再需要频繁跳出平台
2. 工作台开始真正承接 incident

### 阶段 3：Workflow 联动

目标：让 incident 可触发自动化和 SOP。

交付：

1. incident 触发 workflow
2. 手动触发 runbook
3. 审批节点
4. 执行日志
5. 失败处理

状态标准：

1. 至少 2 到 3 个标准运维动作可被稳定触发

### 阶段 4：Terminal 联动

目标：让 incident 可以进入受控执行面。

交付：

1. incident 上下文注入 terminal
2. 与线程 / sandbox 绑定
3. 高风险命令限制
4. 执行记录与审计回流

状态标准：

1. 事件到执行形成闭环

## 7. 具体实施顺序建议

### 7.1 第一周到第二周

1. 定义数据模型
2. 建库建表
3. 完成 webhook ingest
4. 完成 incident 列表接口

### 7.2 第三周到第四周

1. 完成 Alertmanager 接入
2. 完成 signal 标准化
3. 完成 incident 聚合逻辑
4. 完成事件详情页

### 7.3 第五周到第六周

1. 接入 AI 摘要
2. 接入历史上下文
3. 加入工作台入口
4. 补权限和审计

### 7.4 第七周到第八周

1. 联动 workflow
2. 联动 terminal
3. 做端到端回归
4. 准备第一批真实场景试点

---

## 8. 当前最适合的开发方式

### 8.1 不建议先拆独立微服务

当前阶段建议先以内嵌方式接入 DeerFlow 主平台。

原因：

1. 产品仍在验证期
2. 需要快速和工作台打通
3. 需要复用当前用户与权限体系
4. 现在拆服务会显著增加工程复杂度

### 8.2 当前推荐方式

第一阶段建议：

1. 后端新增 router / service / repository
2. 前端新增智能告警页面
3. 数据库新增 incident 相关表
4. 先在当前平台内联运行

### 8.3 何时再考虑拆服务

出现以下情况时再考虑：

1. 告警吞吐明显上升
2. ingest 链路需要独立扩容
3. 异步处理变重
4. 不同团队需要独立部署告警能力

---

## 9. 里程碑验收标准

### M1：接入可用

1. 外部告警可以稳定进入平台
2. 支持 webhook / Alertmanager

### M2：事件可用

1. 多条告警能归并成 incident
2. 事件列表可筛选、可查看详情

### M3：AI 可用

1. incident 可生成摘要
2. 用户能快速理解问题

### M4：工作台可用

1. incident 可进入 DeerFlow 工作台
2. 用户能基于事件继续协作和处置

### M5：执行可用

1. incident 可触发工作流
2. incident 可进入终端执行上下文

---

## 10. 最终建议

对 DeerFlow 来说，智能告警最好的启动方式不是做“大而全告警平台”，而是：

**先做一个嵌入 DeerFlow 的 Incident MVP，再逐步向工作台、工作流和终端延展。**

这条路线既能借鉴业界成熟产品的设计，又不会被它们的重型结构拖住。
