下面是一份重新汇总后的方案。核心结论很明确：**前端 UI 保持现状，智能终端能力迁移到我们的后端 LangGraph 架构中实现；Chaterm 只作为能力参考，不复制它的 agent 框架。**

**总体定位**
智能终端不是一个前端组件功能，而应该是 OpsinTech 平台级 Agent 能力。

现有前端 `terminal-agent-ui.tsx` 的体验可以保留，包括：

- Command / Agent 模式切换
- 命令确认卡片
- 命令执行结果展示
- 危险操作提示
- 交互输入框
- Todo / 任务状态展示
- 会话历史入口
- 安全设置入口

需要重构的是底层架构：把当前分散在前端、Next API、FastAPI router 里的 agent loop、命令执行、安全判断、交互检测，统一迁移到后端 LangGraph。

**目标架构**
```text
Frontend Terminal Agent UI
  terminal-agent-ui.tsx
  useTerminalAgent.ts
        |
        v
Terminal Agent API / Streaming API
        |
        v
LangGraph Terminal Agent
        |
        +-- Intent / Plan Node
        +-- Tool Selection Node
        +-- Command Security Node
        +-- Approval Node
        +-- Terminal Execution Node
        +-- Interaction Detection Node
        +-- Observation Node
        +-- Final Response Node
        |
        v
Terminal Session Service
        |
        v
SSH / PTY Session
```

**核心原则**
1. **使用我们的 LangGraph 架构**
   - 不把 Chaterm 的 `Controller/Task/Agent Loop` 原样搬过来。
   - 用 LangGraph 编排智能终端状态流。

2. **前端 UI 保持不变**
   - 当前 UI 方向是合理的。
   - 前端只做展示、确认、输入、状态同步。
   - 不承担最终安全判断和命令执行逻辑。

3. **后端是唯一可信边界**
   - 命令执行、安全检查、资产权限、租户隔离、审计日志都在后端完成。
   - 前端传来的任何命令都要重新校验。

4. **Agent 和用户终端共享同一个 Session**
   - 不能再让 UI 终端走 WebSocket，而 Agent 模式另开 `/terminal/execute` SSH。
   - 否则 `cd`、环境变量、sudo 状态、交互上下文都会不一致。

5. **Command 模式必须严格确认**
   - 所有远端动作都必须等待用户确认。
   - 不只是 `execute_command`，`read_file/write_file/grep/glob/web_fetch` 这类工具也不能绕过确认。

6. **Agent 模式是有限自治**
   - 安全命令可以自动执行。
   - 高风险/破坏性命令进入人工确认。
   - 被阻止的命令不能让模型绕过或改写继续执行。

**后端方案**
Phase 1 后端：Terminal Session Service

建立统一终端会话服务：

- `create_session(asset_id, user_id, tenant_id)`
- `attach_websocket(session_id)`
- `execute_command(session_id, command)`
- `send_input(session_id, input)`
- `resize(session_id, rows, cols)`
- `cancel_command(session_id, command_id)`
- `close_session(session_id)`

职责：

- 管理 SSH / PTY 生命周期。
- 维护当前 session 状态。
- 支持 UI 终端和 Agent 共用同一连接。
- 负责资源释放、断线恢复、超时处理。

Phase 2 后端：Terminal Command Runner

实现稳定命令执行器：

- 用 start/end marker 包裹命令。
- 捕获 stdout/stderr/exit code。
- 支持 timeout。
- 支持 cancel。
- 支持 streaming output。
- 支持半行输出、ANSI 清理、命令 echo 过滤。
- 替代当前 prompt regex 捕获方式。

参考 Chaterm 的 marker-based runner 思路，但按 Python/FastAPI/PTY 体系重新实现。

Phase 3 后端：Command Security Service

统一安全检查：

- `allow`
- `ask`
- `block`

检查维度：

- 危险命令：`rm`, `dd`, `mkfs`, `reboot`, `shutdown`, `chmod`, `chown`, `systemctl`, `iptables` 等。
- 黑名单模式：`rm -rf /`, `curl | sh`, fork bomb 等。
- 复合命令：`;`, `&&`, `||`, `|`。
- 用户权限。
- 租户策略。
- 资产策略。
- Command / Agent 模式策略。
- 自动批准策略。

所有执行都必须经过它，包括 Agent 自动执行。

Phase 4 后端：LangGraph Terminal Agent

新增 Terminal Agent Graph。

建议节点：

```text
User Input
   |
Intent / Plan Node
   |
Tool Selection Node
   |
Command Security Node
   |
   +-- block --> Final Response Node
   |
   +-- ask --> Approval Node
   |
   +-- allow --> Terminal Execution Node
                    |
                    v
            Interaction Detection Node
                    |
                    v
              Observation Node
                    |
                    +-- continue --> Tool Selection Node
                    |
                    +-- done --> Final Response Node
```

Graph State 建议包含：

- `thread_id`
- `terminal_session_id`
- `asset_id`
- `tenant_id`
- `user_id`
- `mode`: `cmd | agent`
- `messages`
- `todos`
- `current_plan`
- `pending_tool_call`
- `pending_approval`
- `command_history`
- `security_results`
- `interaction_state`
- `final_response`

Phase 5 后端：Interaction Detection Service

交互检测迁移到后端执行流中：

支持：

- password / passphrase
- sudo password
- yes/no 确认
- numbered selection
- pager：`less`, `more`, `journalctl`, `systemctl status`
- press enter
- TUI：`vim`, `nano`, `top`, `htop`, `tmux`
- 超时和静默检测

后端检测到交互后返回：

```json
{
  "type": "waiting_input",
  "command_id": "...",
  "interaction_type": "password",
  "prompt_hint": "请输入 sudo 密码",
  "options": []
}
```

前端继续复用当前 `CommandInteractionInput`。

Phase 6 后端：审计与持久化

需要持久化：

- terminal session
- chat session
- command execution record
- approval record
- tool call record
- stdout/stderr 摘要
- security result
- final answer
- user feedback

这样后续可以支持：

- 会话恢复
- 审计回放
- 知识库总结
- 技能沉淀
- 工单审批
- 多资产批量操作

**前端方案**
Phase 1 前端：保留 UI，改通信层

保留：

- `terminal-agent-ui.tsx`
- 命令卡片
- 模式切换
- 安全设置弹窗
- 历史入口
- 交互输入组件

重构：

- `useTerminalAgent.ts`

它应变成一个较薄的 hook：

- `sendMessage`
- `confirmToolCall`
- `rejectToolCall`
- `submitInteraction`
- `cancelCommand`
- `switchMode`
- `subscribeEvents`

前端不再自己决定命令是否危险，只展示后端返回的 `security_result`。

Phase 2 前端：事件协议统一

后端向前端推送统一事件：

```text
assistant_message_delta
tool_call_created
security_check_completed
approval_required
command_started
command_output_delta
interaction_required
command_completed
command_failed
todo_updated
final_response
```

前端按事件更新现有 UI 状态。

Phase 3 前端：Next API 收敛

当前 `frontend/src/app/api/terminal/chat/route.ts` 不应该继续承担完整 agent loop。

可选两种方式：

- 方案 A：废掉，前端直接连 FastAPI Agent API。
- 方案 B：保留为 BFF，只做鉴权 cookie 转发、SSE 转发、兼容前端调用。

不建议在 Next API 里继续维护工具执行逻辑。

**Chaterm 借鉴范围**
可以借鉴：

- marker-based command runner 思路
- interaction detector 规则
- TUI / pager / password 检测经验
- command security parser 思路
- Command / Agent 模式产品体验
- todo/focus 的任务管理思想
- 非交互命令 prompt 约束

不建议照搬：

- Electron main process agent controller
- Chaterm `Task` 生命周期
- IPC 消息体系
- 本地磁盘 storage
- provider abstraction
- 整套 Cline 风格 agent loop

原因：

- 我们已有 LangGraph。
- 我们已有资产、租户、模型配置、后端网关。
- 直接复制会形成第二套 agent 核心。
- Chaterm 是 GPL-3.0，直接复制大量源码有许可风险。

**当前代码需要纠正的问题**
1. `Command` 模式语义不完整  
   当前部分工具仍可能在服务端直接执行，必须改成全部走确认。

2. `Agent` 和 UI 终端执行链路分裂  
   当前 `/terminal/execute` 新开 SSH，不应作为长期方案。

3. 安全检查不统一  
   前端有较完整 checker，但真正执行路径没有统一使用。

4. 交互检测时机不对  
   现在很多情况下是命令结束后才检测输出，无法处理真正阻塞的交互。

5. 会话历史不应只放 localStorage / 内存  
   需要后端持久化。

6. `terminal.py` 目前偏原型  
   需要拆出 service 层，不应把 PTY、SSH、执行、历史全塞在 router 里。

**推荐实施顺序**
1. 后端：抽出 `TerminalSessionService`
2. 后端：实现 marker command runner
3. 后端：实现 `CommandSecurityService`
4. 后端：搭建 `TerminalAgentGraph`
5. 前端：重构 `useTerminalAgent.ts` 通信层
6. 前端：保持 UI，接入后端事件流
7. 后端：接入交互检测
8. 后端：持久化会话、命令、审批、审计
9. 测试：补安全、模式、交互、session 一致性测试

**测试重点**
后端：

- Command 模式所有远端工具都需要确认。
- Agent 模式安全命令自动执行。
- Agent 模式危险命令进入确认。
- block 命令无法执行。
- 复合命令能被正确解析。
- UI terminal 和 Agent 共享 session。
- `cd /tmp` 后 Agent 命令仍在 `/tmp`。
- 命令 timeout 正确释放资源。
- WebSocket 断开清理 PTY。
- 无权限 asset 不能创建 session。

前端：

- 命令确认卡片正常。
- 拒绝命令后 graph 能继续返回说明。
- 危险命令展示安全原因。
- 交互输入框能响应后端 `interaction_required`。
- 命令输出 delta 正确追加。
- 会话刷新后能恢复状态。

集成：

- 用户请求排查服务状态。
- Agent 生成命令。
- 安全检查通过。
- 后端执行。
- 输出回传。
- Agent 继续分析。
- 最终总结。

**最终结论**
是的，应该使用我们的 LangGraph 架构。

最终方向是：

```text
Chaterm = 参考产品和模块经验
OpsinTech LangGraph = 智能终端的大脑
FastAPI Terminal Session = 执行与安全边界
Frontend UI = 交互展示层
```

这样实现出来的智能终端才是 OpsinTech 平台能力，而不是 Chaterm agent 的局部复刻。
