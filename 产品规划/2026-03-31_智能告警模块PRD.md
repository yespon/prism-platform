# 2026-03-31 智能告警模块 PRD

## 1. 文档目标

本文档定义 DeerFlow 智能告警模块的产品目标、用户场景、能力边界、页面结构与 MVP 范围，用于指导后续产品设计与研发实施。

---

## 2. 产品定义

智能告警不是一个“更好看的告警列表”，也不是一个“加了 AI 摘要的 alerta”。

DeerFlow 的智能告警模块应定义为：

**一个把原始告警压缩、理解、组织并升级为可处置事件的智能入口系统。**

它的作用不是停留在“收告警”，而是完成以下四件事：

1. 把噪音变成事件
2. 把事件变成可理解上下文
3. 把上下文流入工作台
4. 把事件升级为流程或执行动作

---

## 3. 背景问题

当前企业在告警处理上通常会遇到以下问题：

1. 告警太多，噪音太大
2. 同一个问题会触发多条告警，但没有形成统一事件
3. 告警字段很多，但缺少真正可操作的上下文
4. 值班人员需要人工拼接日志、指标、变更、责任人信息
5. 告警系统和处置系统是分离的，导致处理链路长
6. 告警的状态流转有，但治理能力不够产品化

传统开源告警产品通常能覆盖：

1. 接入
2. 展示
3. 基础状态管理
4. 部分关联和去重

但很难覆盖：

1. 智能理解
2. 上下文拼装
3. 事件抽象
4. 协同处置入口
5. 权限、审计、多租户治理

## 6. 核心对象模型

智能告警模块从产品上必须区分三类对象：

### 6.1 Raw Alert

原始告警对象，来自外部系统。

示例来源：

1. Alertmanager
2. Grafana
3. 云厂商告警
4. APM
5. 日志平台
6. 自定义 Webhook

### 6.2 Signal

归一化后的信号对象，作为平台内部统一处理结构。

作用：

1. 标准化字段
2. 统一 severity、service、env、labels
3. 生成 dedup key / correlation key

### 6.3 Incident / Event

面向用户真正展示和处理的对象。

它应成为工作台和后续流程的输入对象。

一个事件可以由多条 Raw Alert / Signal 聚合而成。

## 7. 核心能力设计

### 7.1 告警接入

#### 目标

接住外部信号并转成内部统一对象。

#### 能力点

1. Webhook 接入
2. Alertmanager 接入
3. 标准 JSON 事件接入
4. 字段映射配置
5. 接入鉴权与来源校验
6. 幂等处理

#### MVP 范围

1. 先支持 `Webhook`
2. 先支持 `Alertmanager`

### 7.2 告警归一化

#### 目标

把不同来源告警映射到统一语义模型。

#### 能力点

1. source
2. service
3. environment
4. severity
5. status
6. labels
7. fingerprint / dedup key
8. correlation key

### 7.3 去重与聚合

#### 目标

减少重复告警对用户的干扰。

#### 能力点

1. 同 fingerprint 去重
2. 时间窗口聚合
3. 同服务同异常类型聚合
4. 主从告警折叠
5. 告警风暴抑制

### 7.4 事件生成

#### 目标

把告警变成面向用户可处理的事件。

#### 能力点

1. 同类告警合并为事件
2. 事件自动分配优先级
3. 事件生命周期管理
4. 事件状态流转

建议状态：

1. `new`
2. `triaging`
3. `in_progress`
4. `mitigated`
5. `resolved`
6. `closed`
7. `suppressed`

### 7.5 AI 摘要与解释

#### 目标

让用户快速理解问题。

#### 能力点

1. 一句话总结发生了什么
2. 一句话说明影响范围
3. 一句话说明建议先看哪里

输出示例：

1. 当前支付服务在生产环境出现错误率异常升高，过去 8 分钟内已触发 17 条同类告警。
2. 初步影响为订单支付失败率上升，可能影响华东区在线交易。
3. 建议优先查看最近发布记录、支付网关依赖、应用错误日志与数据库连接池状态。

### 7.6 上下文拼装

#### 目标

让事件详情页不是“告警字段面板”，而是真正可工作的上下文页。

#### 能力点

1. 最近相似事件
2. 最近变更记录
3. 责任团队
4. 关联服务
5. 关联环境
6. 标签解释
7. 时间线
8. 附加建议

### 7.7 事件详情页

#### 目标

提供智能告警的主工作入口。

#### 页面结构建议

1. 顶部事件摘要区
2. 事件基础信息区
3. AI 总结与建议区
4. 关联告警列表
5. 时间线
6. 上下文信息区
7. 操作区

操作区至少包括：

1. 进入工作台
2. 指派负责人
3. 标记误报
4. 屏蔽
5. 升级为流程
6. 打开终端

### 7.8 治理能力

#### 目标

让智能告警成为产品，而不是内部工具。

#### 能力点

1. 多租户隔离
2. 权限控制
3. 审计日志
4. 告警源管理
5. 路由规则管理
6. 屏蔽/静默策略
7. 团队归属与指派

## 8. 典型用户流程

### 8.1 值班处理流程

1. 外部告警进入平台
2. 平台归一化并尝试聚合
3. 平台生成事件
4. AI 自动生成摘要与建议
5. 值班人员查看事件详情
6. 值班人员进入工作台
7. 发起流程或终端执行

### 8.2 平台治理流程

1. 管理员接入新的告警源
2. 配置映射规则
3. 配置静默/聚合策略
4. 观察噪音下降情况
5. 调整规则与优先级

## 10. MVP 边界

### 10.1 MVP 必须做

1. Webhook + Alertmanager 接入
2. 统一事件模型
3. 去重与基础聚合
4. 事件列表页
5. 事件详情页
6. AI 摘要
7. 最近相似事件和简单上下文拼装
8. 进入工作台的入口
9. 基础状态流转
10. 多租户、权限、审计

### 10.2 MVP 暂不做

1. 高级异常检测
2. 海量依赖拓扑根因图
3. 自动化处置全闭环
4. 完整 on-call 体系
5. 大量第三方告警源深度适配
6. 复杂 BI 看板

## 11. 成功指标

### 11.1 产品指标

1. 告警压缩率
2. 事件聚合率
3. 值班首响时间下降
4. 事件进入工作台比例
5. 误报标记率

### 11.2 体验指标

1. 用户是否能在 30 秒内理解事件
2. 用户是否能在 1 分钟内进入处置动作
3. 用户是否能明显减少多系统跳转

## 12. 与其他模块的关系

### 12.1 与智能工作台

智能告警负责把“问题”变成“可理解事件”，工作台负责把“事件”变成“可协作任务”。

### 12.2 与智能工作流

智能告警只负责发现和升级，不负责承载复杂执行编排。

### 12.3 与智能终端

智能告警可以触发终端入口，但不直接等于执行面。

---

## 13. 当前阶段建议

当前最适合 DeerFlow 的落地方式是：

1. 把智能告警先做成事件引擎
2. 用 AI 摘要和上下文拼装打出第一波产品差异化
3. 用工作台承接后续协作和处置

这样做可以最大程度避免把产品再次做成“二开 alerta”。
