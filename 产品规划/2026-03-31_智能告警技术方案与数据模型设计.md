# 2026-03-31 智能告警技术方案与数据模型设计

## 1. 文档目标

本文档用于定义 DeerFlow 智能告警模块的技术实现方案，重点覆盖：

1. 整体架构
2. 核心数据模型
3. 处理链路
4. AI 能力接入点
5. 与工作台、工作流、终端的接口关系
6. MVP 实施建议

---

## 2. 技术目标

智能告警模块在技术上要解决的不是“如何展示告警”，而是：

1. 如何高可靠接收多源告警
2. 如何把异构告警标准化
3. 如何把重复告警压缩成事件
4. 如何为 AI 提供可解释上下文
5. 如何让事件稳定流入工作台和执行链路
6. 如何在多租户、权限、审计约束下运行

---

## 3. 总体架构

建议将智能告警模块拆为六个技术层：

### 3.1 Ingestion Layer

负责接收外部告警输入。

输入来源：

1. Alertmanager
2. Webhook
3. Grafana 告警
4. 云平台告警
5. APM / 日志平台

输出：

1. 原始告警入库
2. 投递到标准化处理链

### 3.2 Normalization Layer

负责把不同来源数据映射为统一的内部 `signal`。

核心动作：

1. 字段映射
2. 时间标准化
3. severity 标准化
4. service / env / source 标准化
5. 去重键生成
6. 关联键生成

### 3.3 Correlation Layer

负责把 signal 收敛成 incident。

核心动作：

1. dedup
2. aggregation
3. suppression
4. correlation
5. incident lifecycle update

### 3.4 Context Layer

负责为 incident 补上下文。

上下文来源：

1. 历史同类事件
2. 最近变更记录
3. 服务目录 / 责任团队
4. 环境信息
5. 用户标签
6. 相关运行数据摘要

### 3.5 AI Triage Layer

负责事件解释与建议。

主要输出：

1. 事件摘要
2. 影响范围说明
3. 排查建议
4. 优先级重估建议

### 3.6 Action Layer

负责把事件挂接到下游能力。

包括：

1. 进入工作台
2. 触发工作流
3. 打开终端上下文
4. 创建任务 / 工单

---

## 4. 核心处理链路

建议处理链路如下：

1. 接收到原始告警 payload
2. 生成 `raw_alert`
3. 标准化为 `signal`
4. 计算 dedup key 和 correlation key
5. 查找是否存在可复用 incident
6. 若存在则归并，否则创建新 incident
7. 更新 incident 状态与统计信息
8. 异步拼装上下文
9. 异步触发 AI 摘要
10. 推送到工作台 / 列表页
11. 根据策略决定是否自动触发工作流或通知

---

## 5. 数据模型设计

### 5.1 模型原则

1. 明确区分原始告警和用户处理对象
2. 所有数据带 `tenant_id / user_scope`
3. 支持幂等接入
4. 支持审计
5. 支持后续扩展到更多规则和 AI 输出

### 5.2 `alert_sources`

用于记录告警接入源。

建议字段：

1. `id`
2. `tenant_id`
3. `name`
4. `type`
5. `status`
6. `auth_mode`
7. `config_payload`
8. `created_at`
9. `updated_at`

### 5.3 `raw_alerts`

用于保存原始输入，作为事实层。

建议字段：

1. `id`
2. `tenant_id`
3. `source_id`
4. `external_event_id`
5. `received_at`
6. `payload_json`
7. `payload_hash`
8. `ingest_status`
9. `error_message`

### 5.4 `signals`

用于保存标准化后的平台内部信号。

建议字段：

1. `id`
2. `tenant_id`
3. `raw_alert_id`
4. `source`
5. `service`
6. `environment`
7. `severity`
8. `status`
9. `title`
10. `summary`
11. `labels_json`
12. `fingerprint`
13. `correlation_key`
14. `occurred_at`
15. `resolved_at`
16. `created_at`

### 5.5 `incidents`

用户真正看到和处理的对象。

建议字段：

1. `id`
2. `tenant_id`
3. `incident_key`
4. `title`
5. `summary`
6. `severity`
7. `priority`
8. `status`
9. `service`
10. `environment`
11. `owner_user_id`
12. `owner_team_id`
13. `signal_count`
14. `first_seen_at`
15. `last_seen_at`
16. `resolved_at`
17. `suppressed`
18. `ai_summary`
19. `ai_impact`
20. `ai_suggestion`
21. `created_at`
22. `updated_at`

### 5.6 `incident_signal_links`

用于关联 incident 和 signals。

建议字段：

1. `id`
2. `tenant_id`
3. `incident_id`
4. `signal_id`
5. `linked_at`

### 5.7 `incident_context_snapshots`

用于记录 AI 和上下文拼装结果，避免页面实时拼装过重。

建议字段：

1. `id`
2. `tenant_id`
3. `incident_id`
4. `context_json`
5. `version`
6. `generated_at`

### 5.8 `incident_actions`

用于记录用户与系统对 incident 的操作。

建议字段：

1. `id`
2. `tenant_id`
3. `incident_id`
4. `actor_id`
5. `action_type`
6. `action_payload`
7. `created_at`

### 5.9 `alert_rules`

用于管理聚合、静默、升级规则。

建议字段：

1. `id`
2. `tenant_id`
3. `name`
4. `rule_type`
5. `enabled`
6. `condition_json`
7. `action_json`
8. `created_at`
9. `updated_at`

## 6. 关键键值设计

### 6.1 Dedup Key

用于识别同一类重复信号。

建议组成：

1. source
2. service
3. environment
4. normalized alert type
5. stable label subset

### 6.2 Correlation Key

用于聚合不同 signal 到同一 incident。

建议组成：

1. service
2. environment
3. incident family
4. time window
5. optional topology / dependency hints

### 6.3 Incident Key

用于标识一个可追踪事件。

建议：

1. 对外展示短 ID
2. 对内使用稳定 UUID

---

## 7. AI 能力接入点

### 7.1 AI 摘要输入

建议输入包括：

1. incident title
2. severity / priority
3. service / environment
4. top signals
5. recent incident history
6. optional deploy/change summary

### 7.2 AI 输出结构

建议不要只保存自由文本，而是保存结构化输出：

1. `summary`
2. `impact`
3. `recommended_next_steps`
4. `possible_related_changes`
5. `confidence`

### 7.3 触发时机

建议异步触发：

1. 新 incident 创建后
2. incident 聚合数量显著变化后
3. 用户主动点击“重新分析”后

### 7.4 风险控制

AI 输出只用于：

1. 解释
2. 建议
3. 辅助优先级判断

在 MVP 阶段不应直接触发高风险执行动作。

## 9. 多租户与安全要求

### 9.1 数据隔离

以下对象必须按租户隔离：

1. alert sources
2. raw alerts
3. signals
4. incidents
5. incident context
6. incident actions
7. rules

### 9.2 权限

建议至少区分：

1. viewer
2. operator
3. manager
4. admin

### 9.3 审计

必须记录：

1. 谁接入了告警源
2. 谁修改了规则
3. 谁指派了事件
4. 谁抑制了事件
5. 谁触发了工作流
6. 谁进入了终端执行

---

## 10. 与其他模块的技术接口

### 10.1 与智能工作台

incident 是工作台的上游对象。

需要提供：

1. incident 基本信息
2. context snapshot
3. AI 摘要
4. action timeline

### 10.2 与智能工作流

incident 可以作为工作流的触发条件。

需要提供：

1. incident id
2. incident severity
3. service / env
4. owner
5. current context

### 10.3 与智能终端

incident 进入终端时，需要带入：

1. incident 关联上下文
2. service / env
3. related hints
4. permission scope

## 11. MVP 实施建议

### 11.1 第一阶段

目标：把“告警箱”变成“事件引擎”。

建议实现：

1. 告警接入
2. raw_alert -> signal 标准化
3. dedup / 基础 aggregation
4. incident 创建
5. incident 列表 / 详情页
6. AI 摘要

### 11.2 第二阶段

目标：把事件引擎变成“处置入口”。

建议实现：

1. context snapshot
2. 历史事件关联
3. 工作台联动
4. 操作审计
5. 指派和状态流转

### 11.3 第三阶段

目标：把处置入口变成“执行入口”。

建议实现：

1. 工作流触发
2. 终端联动
3. 规则引擎增强
4. 优先级重估

---

## 12. 技术风险与注意事项

### 12.1 风险一：过早做复杂算法

如果一开始把重点放在异常检测、拓扑根因、图算法，会让项目过重。

MVP 应优先把事件抽象和上下文拼装做出来。

### 12.2 风险二：只有 AI 摘要，没有结构化事件模型

如果没有 `raw_alert / signal / incident` 三层模型，后续工作台和工作流会非常难接。

### 12.3 风险三：忽略治理能力

如果没有租户、权限、审计，后面每新增一个执行能力都会放大风险。

### 12.4 风险四：告警页和工作台边界不清

建议坚持：

1. 告警模块负责事件形成与理解
2. 工作台负责协作与处置

---

## 13. 最终建议

DeerFlow 的智能告警模块最值得做的，不是成为另一个开源告警系统，而是成为：

**整个智能运维平台的事件理解入口层。**

从技术上，最重要的不是先做最复杂的 AI，而是先把：

1. 数据模型
2. 事件抽象
3. 上下文拼装
4. 多租户治理
5. 与工作台/工作流/终端的接口

这几件事做正确。
